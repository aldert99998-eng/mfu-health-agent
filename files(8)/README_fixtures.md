# Fixtures для автотестирования ИИ-агента МФУ

**Источник:** реальная Zabbix-выгрузка `_history_text_SELECT_*.json` (107 846 строк, 68 устройств, диапазон 2025-12-23 — 2026-01-22).

**После нормализации:** 8 734 события ошибок на 43 устройствах.

**Формат:** CSV с колонками `device_id, device_serial, model, timestamp, error_code, error_description, mileage, toner_level, drum_level, fault_history`.

### 12_duplicates_and_code_formats.csv — Дубликаты + форматы кодов

**Назначение:** проверка нормализации кодов ошибок и обработки дубликатов.
**Размер:** 70 строк:
- 10 устройств × 2 события × 2 (точные дубликаты) = 40
- 10 устройств × 3 варианта одного кода (`75-530-00`, `75 530 00`, `75_530_00`) = 30
**Тест-кейсы:** TC-E-030 (нормализация кодов), TC-E-027 (дубликаты), TC-E-057 (count_repetitions).

**Ожидание на выходе:**
- Все три формата `75-530-00`, `75 530 00`, `75_530_00` → нормализованы в единый канонический `75-530-00`.
- Дубликаты: поведение согласно спецификации:
  - Если включена дедупликация — в factor_store 40 уникальных событий вместо 70.
  - Если нет — 70 событий, но `count_error_repetitions` корректно считает повторы.
- Для первого устройства (MFP-00077): `count_error_repetitions('75-530-00', window_days=30)` = 3 (три варианта формата, схлопнутые в один код).
- В отчёте ingestion — предупреждение: `duplicates_detected: 20 rows`.

## Как использовать эти fixtures

### Для ручного прогона

1. Положить все файлы в `tests/fixtures/` проекта.
2. Через UI Streamlit (страница Dashboard) загружать по одному.
3. Сверять результат с ожиданиями из этого README.

### Для автотестов (pytest)

```python
# tests/conftest.py
FIXTURES_DIR = Path(__file__).parent / "fixtures"

@pytest.fixture
def smoke_csv():
    return FIXTURES_DIR / "01_valid_smoke.csv"

@pytest.fixture
def golden_csv():
    return FIXTURES_DIR / "05_golden_reference.csv"

# tests/test_ingestion.py
def test_smoke_ingestion(smoke_csv):
    result = ingestion_pipeline.run(smoke_csv)
    assert result.rows_processed == 50
    assert len(result.factor_store.list_devices()) == 10
    assert result.warnings == []

def test_missing_device_id_rejected():
    path = FIXTURES_DIR / "09_missing_device_id.csv"
    with pytest.raises(MissingRequiredFieldError) as exc:
        ingestion_pipeline.run(path)
    assert "device_id" in str(exc.value)
```

### Для QA-агента (Claude Code + Playwright MCP)

```
Используй QA-промт из 10_qa_agent_prompts.md в режиме EXECUTE.
Fixtures лежат в tests/fixtures/. 

Прогонка:
1. Для каждого файла из README_fixtures.md — загрузить через UI, 
   сверить фактический результат с "Ожидание на выходе".
2. Если результат не совпадает — TC FAIL с деталями.
3. Итоговый отчёт — 12 PASS/FAIL + сводка.
```

## Ограничения текущего набора

- **Нет ресурсных данных отдельно** — tone_level/drum_level/mileage синтезированы детерминированно из device_id, а не взяты из Zabbix (в исходнике только события). Это не искажает логику тестов, но учти.
- **Все устройства Xerox/Lexmark/Ricoh** — в исходных данных нет HP, Canon, Kyocera. Если агент использует `model_aliases.yaml` для этих брендов — нужна отдельная fixture.
- **Диапазон дат декабрь 2025 — январь 2026** — при тестировании формулы A (фактор возраста) учти актуальную дату; если проверяешь через 3 месяца, все события уже «старые».

## Хеш-суммы для проверки целостности

После генерации зафиксируй:

```bash
cd tests/fixtures
sha256sum *.csv > checksums.sha256
```

Любое расхождение в CI — значит fixture случайно изменился, нужно пересчитать expected результаты.
