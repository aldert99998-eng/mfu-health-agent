# Трек D — RAG-база

## Назначение и границы ответственности

Этот трек описывает **подсистему Retrieval-Augmented Generation**: хранение сервисной документации, векторно-лексический поиск, реранкинг, оценку качества и административные операции (индексация, переиндексация, очистка).

### Что входит в трек

- Структура коллекций в Qdrant
- Схема payload'а (метаданные) для чанков
- Стратегии чанкинга по типам документов
- Hybrid search: dense + sparse с RRF-слиянием
- Reranker: cross-encoder поверх top-K кандидатов
- Pipeline индексации (парсинг → чанкинг → обогащение → эмбеддинги → upsert)
- Переиндексация, очистка, экспорт
- Eval-датасет и метрики качества
- UI операций на странице 4 админ-панели

### Что НЕ входит в трек

- Логика расчёта индекса здоровья (трек A)
- Агент и его инструменты (трек B) — но `search_service_docs` и `classify_error_severity` из трека B реализуются в этом слое
- Структура отчёта (трек C)
- Формат входного файла мониторинга (трек E)

RAG — **домен-знания агента**. Агент обращается к этому слою через два инструмента, описанных в треке B. Внутреннее устройство слоя скрыто за этим интерфейсом.

---

## Контекст — что этот трек задаёт для остальных

- **Для трека A:** формат возврата `classify_error_severity` — структура с `severity`, `affected_components`, `source`. Поля `affected_components` напрямую используются в формуле C (правила применения модификаторов).
- **Для трека B:** реализация двух tool-функций. Tool-вызовы агента приземляются сюда.
- **Для трека C:** поле `doc_references` в паттернах и карточках устройств собирается из результатов RAG (ссылки на разделы документации).
- **Для трека E:** после нормализации модели устройства — имя модели приходит в RAG как payload-фильтр. Несоответствие имён приводит к промахам поиска.

---

## Ключевой принцип проектирования

**Метаданные в payload'е Qdrant проектируются от запросов, а не от документов.**

Мы уже знаем, какие фильтры агент будет использовать — `model`, `error_code`, `content_type`. Значит структуру payload'а определяют именно эти поля. Всё остальное — опциональное обогащение.

Этот принцип отличает работающий RAG от «проиндексировали и надеемся».

## Детальное описание компонентов

### 1. Коллекции в Qdrant

Четыре коллекции, каждая — отдельный индекс. Разделение даёт:
- Свою стратегию чанкинга на коллекцию.
- Независимые метрики качества.
- Переиндексацию только нужной части при обновлении.
- Payload-фильтрацию по типу документа на стороне агента.

| Коллекция | Что содержит | Обязательность в демо |
|---|---|---|
| `service_manuals` | Сервисные руководства по моделям МФУ | Обязательно |
| `error_codes` | Справочники кодов ошибок (таблицы код→действие) | Желательно |
| `internal_guides` | Внутренние инструкции инженеров | Опционально |
| `parts_catalog` | Каталог ЗИП | Опционально |

Архитектура позволяет добавлять новые коллекции без изменения кода — достаточно записи в `rag_config.yaml`.

### 2. Схема payload'а

Единая схема для всех коллекций (часть полей может быть `null`).

```python
{
    # Идентификация
    "chunk_id": "sm_kyocera_3253_p142_c2",
    "document_id": "sm_kyocera_3253",
    "collection": "service_manuals",
    
    # Привязка к оборудованию
    "vendor": "Kyocera",
    "models": ["TASKalfa 3253ci", "TASKalfa 3252ci"],
    "model_family": "TASKalfa 3000 series",
    
    # Структура документа
    "doc_title": "Service Manual TASKalfa 3253ci",
    "doc_version": "Rev.3 2023",
    "section": "7.2 Fuser unit troubleshooting",
    "page_number": 142,
    
    # Тип и содержание
    "content_type": "procedure",
    "error_codes": ["C6000", "C6020"],
    "components": ["fuser", "heat_roller"],
    
    # Текст
    "text": "Полный текст чанка",
    "language": "ru",
    
    # Служебное
    "indexed_at": "2026-04-18T10:00:00Z",
    "chunk_size_tokens": 412,
    "embedding_model": "bge-m3",
    "source_hash": "sha256:ab3..."
}
```

**Критические поля для работы агента:**

**`models` (array).** Один чанк может относиться к нескольким моделям (производители часто делают общие манулы на серию). Qdrant фильтрует массивы оператором `match any`, это работает нативно.

**`error_codes` (array).** Коды извлекаются регуляркой на этапе индексации. Паттерны: `[CJEF]\d{3,5}`, `SC\d{3}`, и т.д. — список под каждого вендора. Без этого поля поиск по коду работает только через эмбеддинги, что неустойчиво: BGE-M3 плохо различает близкие коды (`C6000` vs `C6020` — почти идентичные токены).

**`content_type` (enum).** Категория содержимого. Пять значений:

| content_type | Что это | Когда важен |
|---|---|---|
| `symptom` | Описание проявления ошибки | Поиск «что видит пользователь» |
| `cause` | Описание причины | Анализ корневой причины |
| `procedure` | Пошаговые действия | Ответ на «как починить» |
| `specification` | Технические характеристики, допуски | Валидация показаний датчиков |
| `reference` | Таблицы, справочники, коды | Поиск по конкретному коду |

Классификация делается LLM на этапе индексации — один батч-вызов на 10–20 чанков.

**`components` (array).** Узлы, упомянутые в чанке: `fuser`, `drum`, `scanner`, `fan`, `laser_unit`. Извлекаются по словарю синонимов с русско-английским отображением. Это поле используется `classify_error_severity` для возврата `affected_components`.

### 3. Индексы в Qdrant

Для эффективной фильтрации на payload-полях создаются индексы при инициализации коллекции:

```python
client.create_payload_index("service_manuals", "models", field_schema="keyword")
client.create_payload_index("service_manuals", "vendor", field_schema="keyword")
client.create_payload_index("service_manuals", "error_codes", field_schema="keyword")
client.create_payload_index("service_manuals", "content_type", field_schema="keyword")
client.create_payload_index("service_manuals", "components", field_schema="keyword")
client.create_payload_index("service_manuals", "document_id", field_schema="keyword")
```

Без payload-индексов фильтры деградируют до линейного скана — на больших коллекциях это ощутимо медленнее.

### 4. Стратегии чанкинга

Каждая коллекция — своя стратегия.

#### 4.1 `service_manuals` — иерархический структурный

Сервисное руководство имеет чёткую структуру разделов. Используем её, не нарезаем вслепую.

Процесс:
1. Извлекаем оглавление (из PDF bookmarks, fallback — парсинг заголовков по шрифту и выравниванию).
2. Разбиваем документ на **секции** по заголовкам уровня 2–3.
3. Если секция > 800 токенов — рекурсивно делим по параграфам с перекрытием 100 токенов.
4. Если секция < 200 токенов — оставляем как есть, не объединяем с соседями.

Мотивация: раздел «Troubleshooting fuser» остаётся семантически цельным, даже если 600 токенов. При слепой нарезке по 512 разрыв попадает в середину процедуры, оба чанка становятся бесполезными.

#### 4.2 `error_codes` — один чанк = одна запись

Справочники кодов — табличные. Каждая запись атомарна. Чанкуем по строкам:

```
Код C6000.
Симптом: устройство не выходит из режима прогрева.
Причина: отказ термистора фьюзера или его обрыв.
Действия: проверить разъём CN-3, измерить сопротивление термистора (см. табл. 4).
Запчасти: термистор 302KV94100, фьюзер в сборе 302KV93025.
```

Средний размер — 80–150 токенов.

#### 4.3 `internal_guides` — рекурсивный по размеру

Обычно нет жёсткой структуры. Классика: `RecursiveCharacterTextSplitter` с размером 500 токенов, перекрытием 80. Разделители приоритезируем: параграф → предложение → слово.

#### 4.4 `parts_catalog` — аналогично `error_codes`

Одна запчасть = один чанк.

### 5. Векторы: dense + sparse в одной коллекции

Qdrant поддерживает **named vectors** — несколько векторов на один point.

```python
vectors_config = {
    "dense": VectorParams(
        size=1024,
        distance=Distance.COSINE,
        hnsw_config=HnswConfigDiff(m=16, ef_construct=200)
    ),
}
sparse_vectors_config = {
    "sparse": SparseVectorParams(modifier=Modifier.IDF),
}
```

**Почему BGE-M3.** Эта модель уникальна тем, что **одним прогоном** генерирует и dense, и sparse-векторы. Не нужно держать две отдельные модели (например, отдельно BGE-large для dense + SPLADE для sparse). Для on-prem это критично: меньше ресурсов, одна модель в памяти.

**Параметры HNSW.** `m=16, ef_construct=200` — стандарт для коллекций с < 1М векторов. Для демо с десятками тысяч чанков этого с запасом.

### 6. Hybrid search с RRF-слиянием

Запрос агента запускает **два параллельных поиска** — по dense и по sparse векторам. Результаты объединяются через Reciprocal Rank Fusion.

```python
async def hybrid_search(query: str, filters: dict, top_k: int = 30):
    dense_vec, sparse_vec = embedding_model.encode(query, return_sparse=True)
    
    dense_task = client.search(
        collection_name=collection,
        query_vector=("dense", dense_vec),
        query_filter=filters,
        limit=top_k
    )
    
    sparse_task = client.search(
        collection_name=collection,
        query_vector=("sparse", sparse_vec),
        query_filter=filters,
        limit=top_k
    )
    
    dense_hits, sparse_hits = await asyncio.gather(dense_task, sparse_task)
    
    return rrf_merge(dense_hits, sparse_hits, k=60)
```

Формула RRF:

```
score_rrf(d) = Σ  1 / (k + rank_q(d))
              q∈{dense, sparse}
```

где `k = 60` — стандартная константа, сглаживает влияние ранга.

**Альтернатива — Qdrant Query API с прерасчётом prefetch.** В свежих версиях Qdrant (>= 1.10) есть встроенный `Fusion` mode:

```python
client.query_points(
    collection_name=collection,
    prefetch=[
        Prefetch(query=dense_vec, using="dense", limit=30),
        Prefetch(query=sparse_vec, using="sparse", limit=30),
    ],
    query=FusionQuery(fusion=Fusion.RRF),
    limit=30
)
```

Это предпочтительный вариант — вся работа внутри Qdrant, меньше сетевых вызовов. Используем именно его если версия клиента позволяет.

**Зачем hybrid, а не только dense.** Коды ошибок и номера запчастей плохо ловятся эмбеддингами из-за токенизации. Sparse/BM25-подобный поиск даёт точное лексическое совпадение. Dense — семантическое сходство для запросов вроде «проблема с закреплением тонера». Вместе закрывают оба случая.

### 7. Reranker

После hybrid search у нас top-30 кандидатов. Reranker — cross-encoder, который видит пару (запрос, документ) **одновременно** и выдаёт точный score релевантности.

**Модель:** `BAAI/bge-reranker-v2-m3`. Многоязычный, хорошо работает с русским. Размер ~500M параметров, помещается на CPU.

**Латентность:** 30 документов батчем — 200–500 мс на современном CPU, 50–100 мс на GPU.

```python
def rerank(query: str, candidates: list[ScoredPoint], top_n: int = 8):
    pairs = [(query, c.payload["text"]) for c in candidates]
    scores = reranker_model.compute_score(pairs, batch_size=16)
    
    ranked = sorted(
        zip(candidates, scores),
        key=lambda x: x[1],
        reverse=True
    )
    
    return [c for c, _ in ranked[:top_n]]
```

**Итоговая воронка:**
- Hybrid search: top-30 кандидатов.
- Reranker: top-8 (настраивается).
- В контекст агента: top-5 (настраивается).

### 8. Ingestion pipeline

Пошаговый процесс при загрузке документа через страницу 4 админ-панели.

```
┌─────────────────────────────────────────────────────────┐
│ 1. Парсинг                                              │
│    PDF → текст с сохранением структуры                  │
│    Сохранение hash'а файла                              │
├─────────────────────────────────────────────────────────┤
│ 2. Предобработка                                        │
│    Чистка OCR-артефактов                                │
│    Нормализация пробелов                                │
│    Склейка разорванных предложений                      │
├─────────────────────────────────────────────────────────┤
│ 3. Определение модели и вендора                         │
│    Из имени файла и первой страницы через LLM           │
│    Подтверждение пользователем                          │
├─────────────────────────────────────────────────────────┤
│ 4. Чанкинг                                              │
│    Стратегия по коллекции                               │
├─────────────────────────────────────────────────────────┤
│ 5. Обогащение метаданных                                │
│    - error_codes: регулярками                           │
│    - components: словарь синонимов                      │
│    - content_type: LLM-классификатор (батчем)           │
├─────────────────────────────────────────────────────────┤
│ 6. Эмбеддинги                                           │
│    BGE-M3, батчами по 32                                │
│    Одновременно dense и sparse                          │
├─────────────────────────────────────────────────────────┤
│ 7. Upsert в Qdrant                                      │
│    Батчами по 100 point'ов                              │
│    Прогресс в UI                                        │
└─────────────────────────────────────────────────────────┘
```

**Восстанавливаемость.** Процесс сохраняет checkpoint после каждого этапа. При падении — можно продолжить с последнего успешного.

**Идемпотентность.** Повторный запуск индексации того же документа перезаписывает старые чанки: сначала удаляет все point'ы с `document_id = X`, затем загружает новые. `source_hash` позволяет пропустить индексацию, если файл не менялся.

### 9. Парсинг PDF

Рекомендую комбинированный подход:

- **Первичный слой:** `pymupdf` (fitz) — быстрый, хорошо извлекает текст и bookmarks.
- **Сложные случаи:** `unstructured` — для документов со сложной вёрсткой, таблицами.
- **Сканы / плохой OCR:** опционально — `paddleocr` или `tesseract`. В демо этот путь **не включаем**, только предупреждение «документ, похоже, скан».

### 10. Операции над базой

На странице 4 UI доступны:

#### 10.1 Переиндексация

**Полная.** Удаляем коллекцию, прогоняем ingestion заново по всем документам из `storage/uploads/`. Применяется при:
- Смене модели эмбеддингов.
- Смене стратегии чанкинга.
- Повреждении коллекции.

**Инкрементальная.** Только документы с изменённым `source_hash` + документы, которых нет в Qdrant. Запускается автоматически по расписанию, опционально.

#### 10.2 Очистка

Два варианта:
- Удалить коллекцию целиком (с подтверждением).
- Удалить документы старше N дней (по `indexed_at`).

#### 10.3 Экспорт и импорт

Бэкап коллекции в Qdrant snapshot (`client.create_snapshot`). Используется для миграций между средами.

#### 10.4 Диагностика

Проверка целостности:
- Все ли point'ы имеют payload с обязательными полями.
- Нет ли осиротевших point'ов (`document_id` не существует в `uploads/`).
- Размеры векторов совпадают с конфигом.

### 11. Eval-датасет и метрики

Это часть, которую большинство проектов пропускают. Здесь она обязательная.

#### 11.1 Формат eval-датасета

Файл: `configs/eval_dataset.yaml`.

```yaml
queries:
  - id: "q_001"
    query: "ошибка C6000 на Kyocera TASKalfa 3253ci"
    expected_chunks:
      - chunk_id: "sm_kyocera_3253_p142_c2"
      - chunk_id: "sm_kyocera_3253_p143_c1"
    must_contain_codes: ["C6000"]
    must_contain_components: ["fuser"]
    scenario: "error_code_lookup"
    difficulty: "easy"
  
  - id: "q_002"
    query: "как заменить барабан в Kyocera 3253"
    expected_chunks:
      - chunk_id: "sm_kyocera_3253_p200_c3"
    must_contain_components: ["drum"]
    scenario: "procedure_lookup"
    difficulty: "medium"
  
  - id: "q_003"
    query: "почему устройство пишет что нет тонера хотя он есть"
    expected_chunks:
      - chunk_id: "internal_guides_tonerbug_p1_c1"
    scenario: "symptom_diagnosis"
    difficulty: "hard"
```

Размечается руками на старте (50–100 запросов). Дополняется постепенно реальными вопросами из чата.

#### 11.2 Метрики

**Retrieval-метрики** (по результатам hybrid + reranker, top-K):

| Метрика | Формула | Интерпретация |
|---|---|---|
| Recall@5 | fraction(queries where ≥1 expected ∈ top-5) | Доля запросов, где нужный чанк в топ-5 |
| Recall@10 | fraction(queries where ≥1 expected ∈ top-10) | То же для топ-10 |
| MRR | mean(1 / rank of first relevant) | Средний обратный ранг первого релевантного |
| Precision@5 | mean(|relevant ∩ top-5| / 5) | Доля релевантных в топ-5 |
| nDCG@10 | nDCG с бинарной релевантностью | Учитывает позицию релевантных |

**Per-scenario breakdown.** Все метрики отдельно по каждому `scenario`. Типичная картина: `error_code_lookup` — 95%, `symptom_diagnosis` — 60%. Это даёт прямой сигнал где тюнить.

**Пороги приёмки:**

| Метрика | Приёмлемо | Хорошо | Отлично |
|---|---|---|---|
| Recall@5 (overall) | ≥ 0.70 | ≥ 0.85 | ≥ 0.95 |
| MRR (overall) | ≥ 0.50 | ≥ 0.70 | ≥ 0.85 |

Ниже 0.70 по Recall@5 — RAG считается непригодным для использования, расчёт агента на таких данных даст плохие Severity.

#### 11.3 Статические метрики (по корпусу)

Считаются без eval-датасета, полезны для мониторинга состояния базы:

- Количество документов и чанков.
- Средний/медианный размер чанка (токены).
- Распределение по коллекциям.
- Покрытие моделей МФУ (список моделей в данных vs список моделей, по которым есть документация).
- Дубликаты и near-duplicates (cos-similarity > 0.95 между чанками).
- Средняя длина эмбеддинга до/после нормализации.

#### 11.4 UI: история метрик

На странице 4 — график динамики:
- Recall@5, MRR по времени (точка = запуск eval).
- Отметки на графике: «загружен документ X», «смена модели эмбеддингов», «смена reranker».

Это позволяет видеть, какое действие улучшило или ухудшило поиск. Без этого любая оптимизация RAG — тыкание вслепую.

### 12. LLM для обогащения метаданных

На этапе индексации делаются два типа LLM-вызовов:

**Определение модели и вендора** (1 вызов на документ):
```
Промпт: "Из первой страницы документа определи модель и производителя МФУ.
Верни JSON: {vendor: str, model: str, model_family: str | null}.
Если не уверен — верни null."
```

**Классификация content_type** (батчем на 10-20 чанков):
```
Промпт: "Для каждого чанка ниже отнеси его к одной из категорий:
symptom | cause | procedure | specification | reference.
Верни массив classifications."
```

**Стоимость.** Для документа на 400 страниц с чанкингом получается ~800–1500 чанков, то есть 40–75 батч-вызовов classifier. На локальной LLM — 5–10 минут на документ. Делается один раз при индексации, кэшируется.

**Fallback.** Если LLM недоступна во время индексации — `content_type` ставится в `reference` по умолчанию, документ помечается `needs_llm_enrichment: true`, при следующем запуске индексации дообогащается.

## Риски и крайние случаи

### Риск 1: низкое качество OCR на сканах

**Сценарий:** пользователь загружает сервисный мануал, который на самом деле скан без распознанного текста.
**Защита:** детекция на этапе парсинга — если доля страниц без текстового слоя > 30%, документ помечается `needs_ocr`, индексация блокируется с сообщением. Опционально — включение OCR-пути (не в демо).

### Риск 2: коллекция выросла, поиск стал медленным

**Сценарий:** 500К чанков, запрос занимает > 1 секунды.
**Защита:**
- HNSW параметры — `m` до 24, `ef_construct` до 400.
- Scalar quantization (`int8`) — уменьшает память в 4 раза, почти без потери качества.
- Payload-индексы — обязательны, проверять при росте коллекции.

### Риск 3: реранкер — узкое место

**Сценарий:** 30 документов reranker'а, 500 мс на CPU, это 50% общего времени поиска.
**Защита:**
- Уменьшить `top_n_input` до 15–20 при нехватке производительности.
- Использовать GPU для reranker'а — даёт 5–10x ускорение.
- Опция «только hybrid без reranker» в настройках как fallback.

### Риск 4: разные имена одной модели

**Сценарий:** в документации — «Kyocera TASKalfa 3253ci», в данных мониторинга — «TASKalfa 3253ci» (без бренда) или «3253ci». Payload-фильтр по `model == "TASKalfa 3253ci"` не находит чанки, помеченные как «Kyocera TASKalfa 3253ci».
**Защита:**
- На уровне agent tool: перед RAG-поиском нормализовать имя модели через словарь алиасов (`configs/model_aliases.yaml`).
- Поле `models` в payload — массив, содержит все известные варианты написания.
- При неоднозначности — логирование и проверка на этапе индексации.

### Риск 5: реальный запрос не попадает в eval-датасет

**Сценарий:** eval показывает Recall@5 = 0.9, а пользователи жалуются на плохой поиск.
**Защита:** автоматическое добавление новых запросов из chat-истории в pending-список eval. Инженер раз в неделю размечает их.

### Риск 6: версии моделей эмбеддингов расходятся между индексацией и поиском

**Сценарий:** коллекция индексирована на BGE-M3 v1.0, а поиск запускается с BGE-M3 v1.5 — размерность та же, но семантика сдвинута.
**Защита:** поле `embedding_model` в payload хранит точную версию. При старте проверяется консистентность. Несоответствие блокирует поиск до переиндексации.

### Риск 7: LLM-обогащение вернуло невалидный JSON

**Сценарий:** классификатор content_type выдал текст вместо JSON.
**Защита:** guided JSON output при обогащении. Fallback — content_type = `reference`, флаг `needs_llm_enrichment = true`, повторная попытка в следующую индексацию.

### Риск 8: PII в документации

**Сценарий:** во внутренних guide'ах инженеры оставили контакты «если не работает — звонить +7 999 123-45-67 Иванову».
**Защита:** PII-фильтр на этапе предобработки (раздел 8 pipeline). Regex для телефонов, email, имён. Фильтрация замещающими токенами.

### Риск 9: дубликаты документов

**Сценарий:** один и тот же manual загружен дважды под разными именами. Дублируются чанки, поиск возвращает два одинаковых результата в top-5.
**Защита:** при индексации — проверка `source_hash` против существующих документов в коллекции. Если hash совпадает — предупреждение «такой документ уже есть, перезаписать?».

### Риск 10: обрыв индексации посередине

**Сценарий:** упал процесс в момент индексации большого документа — в Qdrant остались частично загруженные чанки.
**Защита:** индексация — транзакционная на уровне документа. Перед началом — временный флаг `indexing_in_progress: true`. При обнаружении такого флага на старте — удаление partial-чанков, перезапуск с нуля.

### Риск 11: несогласованность affected_components с формулой C

**Сценарий:** RAG возвращает `affected_components: ["фьюзер"]` (по-русски), а правила формулы C используют `fuser` (по-английски).
**Защита:** словарь компонент хранится единым образом в `configs/component_vocabulary.yaml`, используется и при индексации, и при применении правил. Все значения канонические — `fuser`, `drum`, `scanner`.

## Примечание: размер ресурсов

**Для демо-сценария:**

| Ресурс | Размер |
|---|---|
| BGE-M3 модель | ~2.3 ГБ (on disk) |
| BGE-reranker-v2-m3 | ~2.2 ГБ |
| Qdrant с 50К чанков | ~1 ГБ (с векторами 1024-dim) |
| RAM для моделей (fp32) | ~6 ГБ |
| RAM для Qdrant (50К чанков) | ~2 ГБ |

Итого на CPU-инсталляцию: 8–10 ГБ RAM. На GPU с fp16: те же модели в 2 ГБ VRAM каждая.
