# Трек C — Макет HTML и PDF-отчёта

## Назначение и границы ответственности

Этот трек описывает **структуру, визуал и генерацию** итогового отчёта: HTML-версию для просмотра в браузере и PDF-версию для выгрузки и печати.

### Что входит в трек

- Структура из 8 секций отчёта
- Визуальный стиль, цветовая схема, типографика
- Данные модели отчёта (`Report` object)
- Jinja2-шаблон и CSS
- Логика рендеринга HTML → PDF через WeasyPrint
- Обработка крайних случаев (пустой отчёт, 1 устройство, 1000 устройств)
- Interactive-элементы для HTML vs статика для PDF

### Что НЕ входит в трек

- Расчёт индексов (трек A)
- Генерация executive summary и объяснений (делает агент — трек B)
- Кластеризация для топ-5 проблем (инструмент `find_similar_devices` — трек B)
- UI для просмотра отчёта в Streamlit (отдельная задача UI-слоя)

Отчёт — **слой презентации**. Он получает готовые данные и собирает из них документ. Никакой бизнес-логики, никаких LLM-вызовов.

---

## Контекст — что этот трек задаёт для остальных

- **Для трека A:** определяет, какие поля из `HealthResult` попадают в карточку устройства (индекс, confidence, разложение factors, contributions, sources).
- **Для трека B:** требует наличия `executive_summary` (текст, генерируемый агентом отдельным вызовом после batch-прогона) и `pattern_groups` (результат кластеризации для топ-5).
- **Для трека D:** использует источники из RAG (в карточке устройства — ссылка на раздел сервисной документации).
- **Для UI-слоя:** предоставляет `Report.to_html()` и `Report.to_pdf()` как чистые методы.

## Детальное описание компонентов

### 1. Принципы отчёта

**Пирамида.** Самое важное — сверху. Менеджер получает ответ на вопрос «что происходит в парке» из первого экрана, не прокручивая. Инженер проматывает до нужной секции.

**Две аудитории.** Менеджерский материал (executive summary, массовые проблемы, рекомендации) и технический материал (разложение формулы, trace агента, snapshot весов) идут последовательно, не перемешиваются.

**Воспроизводимость.** В отчёте зашит snapshot состояния системы в момент расчёта: профиль весов, версии моделей, дата. Через месяц, глядя на этот PDF, можно однозначно понять, почему индекс получился именно такой.

### 2. Структура — 8 секций

#### Секция 1. Шапка и сводка парка

Компактный блок, занимает верхнюю часть первого экрана (~150–200 px).

| Элемент | Содержимое |
|---|---|
| Заголовок | «Отчёт о здоровье парка МФУ» |
| Подзаголовок | Дата расчёта, имя источника данных, окно анализа |
| Метрика 1 | Средний индекс по парку (большая цифра + дельта к прошлому расчёту) |
| Метрика 2 | Количество в зелёной зоне (зелёный акцент) |
| Метрика 3 | Количество в жёлтой зоне (жёлтый акцент) |
| Метрика 4 | Количество в красной зоне (красный акцент) |
| Индикатор уверенности | Полоска со средним confidence и текстовой интерпретацией |

Четыре плитки-метрики — в горизонтальный ряд, равной ширины.

#### Секция 2. Executive summary

2–3 абзаца связного текста от агента. Не bullet-list, а прочитываемый аналитический текст.

Структура, которую задаёт промпт агента:
- Абзац 1: общая оценка состояния парка.
- Абзац 2: ключевые зоны риска.
- Абзац 3: рекомендации по приоритетам внимания.

Визуально — блок с левой акцентной полоской (border-left), фон чуть светлее основного, шрифт слегка крупнее обычного текста. Выделяется как «вывод», не смешивается со статистикой.

#### Секция 3. Топ-5 проблем

Ключевая секция. Не топ-5 худших устройств, а **топ-5 паттернов**.

Типы паттернов:

| Тип | Критерий | Бейдж |
|---|---|---|
| Массовая проблема | Одна ошибка на 3+ устройствах одной модели | `mass_issue` (коралловый) |
| Локационный паттерн | Концентрация проблем в одной точке | `location_cluster` (фиолетовый) |
| Одиночный критический | H < 20 или Conf < 0.4 | `critical_single` (красный) |

Карточка паттерна:
- Бейдж типа (цветной).
- Заголовок (≤ 60 символов).
- Ключевые числа: затронуто устройств, средний индекс по паттерну.
- Объяснение от агента (2–4 предложения).
- Ссылки на затронутые устройства (якоря на секцию 6).
- Ссылка на раздел документации, если применимо.

Карточки — вертикальный стэк, каждая занимает всю ширину. Между карточками — 16 px.

#### Секция 4. Распределение по парку

Две визуализации бок о бок (или друг под другом на узком экране).

**Визуализация 1 — распределение индексов.**
Гистограмма с 10 бинами (0–9, 10–19, ..., 90–100). Цвет бина соответствует зоне.

**Визуализация 2 — сводка по категории.**
Автовыбор категории для разбивки:
- Если в парке несколько моделей — breakdown по моделям.
- Если одна модель, но несколько локаций — по локациям.
- Иначе — по confidence-зонам.

Горизонтальные bars: название категории, количество устройств, средний индекс.

**Технология:**
- HTML: Plotly.js для интерактивности (hover с деталями).
- PDF: статичные SVG, генерируются через matplotlib или vega-lite с экспортом в SVG.

#### Секция 5. Сводная таблица устройств

Все устройства в одном месте, отсортированы по индексу по возрастанию (худшие сверху).

Колонки:
- ID устройства (с якорной ссылкой на карточку в секции 6)
- Модель
- Локация
- Индекс (число + цветная полоска)
- Confidence (число + индикатор зоны)
- Топ-1 проблема (короткий тег)

В HTML:
- Клик по заголовку колонки — сортировка.
- Поиск по ID/модели/локации через поле фильтрации сверху.
- Фильтры по зоне (кнопки-чипы).
- Виртуализация при большом парке (отображаются только видимые строки).

В PDF:
- Фиксированная сортировка по индексу по возрастанию.
- Повторяющийся заголовок таблицы при переносе на новую страницу (`thead { display: table-header-group }`).

#### Секция 6. Детали по устройствам

Карточка на каждое устройство. Самая большая секция — потенциально сотни карточек.

**Стратегия отображения:**
- HTML: все карточки свёрнуты по умолчанию (`<details>`), разворачиваются кликом.
- PDF: все карточки развёрнуты, компактный дизайн (1 карточка ≈ 0.5 страницы), `page-break-inside: avoid`.

#### Секция 7. Как агент работал (опционально)

Включается флагом `include_agent_trace: true` при генерации отчёта. По умолчанию — выключено (менеджерский отчёт), включено — для технической аудитории.

Содержит:
- Сводка: среднее число tool-вызовов на устройство, среднее число LLM-вызовов, количество перезапусков по self-check, устройства с `flag_for_review`.
- Развёрнутый trace для 2–3 репрезентативных устройств (выбираются автоматически: одно с самым низким индексом, одно с самым высоким, одно с `flag_for_review` если есть).
- Сводка по RAG: количество запросов, средний rerank-score, процент запросов без совпадений.

Trace представляется в читаемом виде:
```
Устройство: MFP-042
├ Шаг 1 (план): Нужно проанализировать события, ресурсы, найти критичность
├ Шаг 2 (tool): get_device_events → 4 события за 30 дней
├ Шаг 3 (tool): get_device_resources → тонер 45%, фьюзер 95%
├ Шаг 4 (tool): classify_error_severity(C6000) → Critical, affected=[fuser]
├ Шаг 5 (tool): count_error_repetitions → 1 раз
├ Шаг 6 (tool): calculate_health_index → H=40, Conf=0.87
└ Шаг 7 (reflection): approved ✓

Всего: 5 tool-вызовов, 3 LLM-вызова, 4200 токенов, 8.3 секунды
```

#### Секция 8. Конфигурация расчёта

Snapshot для воспроизводимости. Выводится компактной таблицей, сгруппированно.

**Группа 1 — профиль весов:**
- Имя профиля, версия
- Severity значения (Critical, High, Medium, Low, Info)
- Параметры R (base, max_value, window_days)
- Параметры C (модификаторы с множителями, max_value)
- Параметры A (tau_days, window_days)
- Пороги зон

**Группа 2 — LLM:**
- Endpoint, имя модели, версия (если доступно)
- Tool-strategy (native / guided_json / react)
- Параметры генерации (temperature и т.д.)

**Группа 3 — RAG:**
- Модель эмбеддингов и её версия
- Модель reranker
- Список использованных коллекций с количеством документов
- Дата последней индексации по каждой коллекции

**Группа 4 — данные:**
- Хэш исходного файла (SHA-256)
- Количество записей на входе / валидных / отбракованных
- Имя профиля маппинга полей (если применён)

### 3. Data model — структура Report

```python
@dataclass
class Report:
    # Метаданные
    report_id: str                       # uuid
    generated_at: datetime
    source_file_name: str
    source_file_hash: str
    analysis_window_days: int
    
    # Сводка
    fleet_summary: FleetSummary
    executive_summary: str               # 2-3 абзаца от агента
    
    # Паттерны
    top_patterns: list[PatternGroup]     # топ-5
    
    # Распределения
    index_distribution: list[DistributionBin]
    category_breakdown: CategoryBreakdown
    
    # Устройства
    devices: list[DeviceReport]
    
    # Технические секции
    agent_trace_summary: AgentTraceSummary | None
    rag_summary: RAGSummary | None
    
    # Snapshot для воспроизводимости
    calculation_snapshot: CalculationSnapshot
    
    # Конфиг рендера
    include_agent_trace: bool = False
    include_devices_collapsed: bool = True


@dataclass
class FleetSummary:
    total_devices: int
    average_index: float
    median_index: float
    zone_counts: dict[str, int]          # {"green": 94, "yellow": 41, "red": 17}
    average_confidence: float
    delta_vs_previous: float | None      # изменение среднего индекса vs предыдущий расчёт


@dataclass
class PatternGroup:
    pattern_type: str                    # "mass_issue" | "location_cluster" | "critical_single"
    title: str
    affected_device_ids: list[str]
    average_index: float
    explanation: str                     # от агента
    doc_references: list[DocReference]


@dataclass
class DeviceReport:
    device_id: str
    model: str | None
    location: str | None
    health_index: int
    confidence: float
    zone: str
    confidence_zone: str
    top_problem_tag: str
    factor_contributions: list[FactorContribution]
    resource_state: ResourceState
    index_history: list[HistoryPoint]    # для спарклайна
    agent_recommendation: str
    flag_for_review: bool


@dataclass
class ResourceState:
    toner: int | None
    drum: int | None
    fuser: int | None
    mileage: int | None
    service_interval: int | None


@dataclass
class DocReference:
    title: str
    section: str
    url: str | None                      # ссылка на внутренний документ
```

### 4. Рендеринг: один шаблон, два вывода

**Ключевой принцип** — один Jinja2 шаблон `report.jinja2` и один CSS-файл, которые используются для обоих форматов.

**Для HTML:**
```python
html = jinja_env.get_template("report.jinja2").render(
    report=report,
    interactive=True        # флаг для шаблона
)
```

**Для PDF:**
```python
html_for_pdf = jinja_env.get_template("report.jinja2").render(
    report=report,
    interactive=False       # статичная версия
)
pdf_bytes = HTML(string=html_for_pdf).write_pdf(
    stylesheets=[CSS("report.css"), CSS("report_print.css")]
)
```

В шаблоне — условия `{% if interactive %}` для переключения между:
- `<details>` vs развёрнутыми блоками
- Plotly.js vs статичный SVG
- Сортируемая таблица vs фиксированная
- Поле поиска vs отсутствие

**Два CSS-файла:**

`report.css` — общие стили (типографика, цвета, отступы, сетка).

`report_print.css` — только для PDF:
```css
@page {
    size: A4;
    margin: 2cm 1.5cm 2cm 1.5cm;
    @top-center {
        content: "Отчёт о здоровье парка МФУ — " string(report-date);
        font-size: 10px;
        color: #666;
    }
    @bottom-right {
        content: counter(page) " / " counter(pages);
        font-size: 10px;
    }
}

h1, h2 { page-break-after: avoid; }
.device-card { page-break-inside: avoid; }
.section { page-break-before: always; }
.section:first-of-type { page-break-before: auto; }

/* Интерактивные элементы скрыты в печати */
.search-bar, .filter-chips, .sort-handles { display: none; }

/* Таблица с повторяющейся шапкой */
.device-table thead { display: table-header-group; }
.device-table tr { page-break-inside: avoid; }
```

### 5. Визуальный стиль

**Типографика:**
- Основной шрифт: Inter, PT Sans, или системный sans-serif.
- Для PDF — обязательно шрифт с полной кириллицей.
- Размеры: h1 24px / h2 18px / h3 16px / body 14px / small 12px.
- Веса: 400 для текста, 500 для заголовков и акцентов. Не использовать 600/700 — утяжеляет.

**Цветовая схема:**

Зоны индекса:
- Зелёная (≥ 75): фон `#e8f5e9`, акцент `#2e7d32`, текст `#1b5e20`
- Жёлтая (40–74): фон `#fff8e1`, акцент `#f57c00`, текст `#e65100`
- Красная (< 40): фон `#ffebee`, акцент `#c62828`, текст `#b71c1c`

Confidence:
- Высокая (≥ 0.85): нейтральный серо-голубой `#eceff1`
- Средняя (0.6–0.85): жёлтая штриховка
- Низкая (< 0.6): красная обводка + плашка «предварительный»

Типы паттернов:
- `mass_issue`: коралловый `#ff7043`
- `location_cluster`: фиолетовый `#7e57c2`
- `critical_single`: красный `#c62828`

**Никаких градиентов** — в PDF через WeasyPrint работают плохо. Только flat fills.

**Отступы:**
- Между секциями: 40 px.
- Внутри секции: 24 px.
- Карточки устройств между собой: 16 px.
- Padding внутри карточки: 20 px.

**Индикаторные полоски:**
Для индекса и confidence — горизонтальные полоски с заливкой:
```html
<div class="bar">
  <div class="bar-fill" style="width: 40%; background: #c62828"></div>
</div>
```
Ширина контейнера = 100 px, высота = 8 px, скругление 2 px.

### 6. Интерактивные элементы (только HTML)

- **Сортировка таблиц** — `<table sortable>` с JS-обработчиком (vanilla JS, ~30 строк).
- **Фильтрация по зонам** — кнопки-чипы сверху таблицы, скрытие строк по data-атрибутам.
- **Поле поиска** — фильтрация по ID/модели/локации, debounce 300 мс.
- **Сворачивание карточек устройств** — `<details>`/`<summary>`, нативно в браузере.
- **Plotly-графики** — hover с деталями, клик по устройству в bar → скролл к карточке.
- **Якорные ссылки** — `#device-MFP-042` для быстрого перехода.

Все JS-скрипты инлайнятся в HTML (не внешние файлы), чтобы HTML-отчёт можно было скинуть одним файлом без зависимостей.

### 7. Обработка крайних случаев

| Случай | Поведение |
|---|---|
| 0 валидных устройств | Секции 3–6 заменяются на плашку «Нет устройств для отображения». Секция 1 показывает нули. |
| 1 устройство | Секция 3 (топ-5 паттернов) — «Слишком мало данных для выявления паттернов». Секция 4 (распределения) — скрыта. |
| 1000+ устройств | Карточки в HTML — только свёрнутые, виртуализация таблицы. В PDF — предупреждение в начале «Отчёт содержит 1000+ устройств, PDF может быть большим». |
| Все устройства здоровы | Секции 3 и 5 показываются, но короче. Executive summary — «Парк в отличном состоянии». |
| Все устройства в красной зоне | Executive summary — «Парк в критическом состоянии». Секция 3 — массовые проблемы доминируют. |
| Нет модели ни у одного устройства | Категорийная разбивка в секции 4 переключается на локации. Если нет и локаций — на confidence-зоны. |
| Очень длинное имя устройства / модели | Обрезается с многоточием в таблице, показывается полностью в карточке. |
| Очень длинный текст объяснения от агента | В карточке — свёрнутый блок «Показать полностью». В PDF — выводится без ограничений. |
| Символы, ломающие CSV/Markdown (кавычки, переносы) | Экранируются Jinja2 автоматически. Для CSV-экспорта (если будет) — отдельная функция escape. |

### 8. Accessibility

- Заголовок страницы `<title>` содержит дату и количество устройств.
- У каждого графика есть `<title>` внутри SVG + `aria-label` на контейнере.
- Цветовая кодировка дублируется текстом («Красная зона» рядом с красной полоской).
- Контрастность текста на цветных фонах — минимум WCAG AA (4.5:1).
- Для таблиц — `<th scope="col">` для доступности скринридеров.

## Интерфейсы модулей

### Основной класс ReportBuilder

```python
class ReportBuilder:
    def __init__(
        self,
        agent: Agent,              # для генерации executive_summary и патернов
        template_dir: Path,
        css_dir: Path
    ):
        ...
    
    def build(
        self,
        health_results: list[HealthResult],
        calculation_snapshot: CalculationSnapshot,
        source_file_info: SourceFileInfo,
        include_agent_trace: bool = False
    ) -> Report:
        """Собирает Report-объект из результатов расчёта."""
        ...
    
    def render_html(self, report: Report) -> str: ...
    
    def render_pdf(self, report: Report) -> bytes: ...
```

### PDF-генератор — отдельный модуль

```python
class PDFGenerator:
    def __init__(self, css_files: list[Path]):
        self.css = [CSS(f) for f in css_files]
    
    def from_html(self, html_string: str, base_url: Path) -> bytes:
        return HTML(string=html_string, base_url=str(base_url)).write_pdf(
            stylesheets=self.css
        )
```

Разделение нужно, чтобы PDF-генератор можно было тестировать отдельно.

## Риски и крайние случаи

### Риск 1: PDF не собирается из-за отсутствующих шрифтов

**Сценарий:** WeasyPrint не находит Inter или PT Sans, откатывается на дефолт — кириллица рендерится неправильно.
**Защита:** шрифты должны быть либо в системе, либо явно подключены через `@font-face` из локального файла. В Docker-образ кладутся .ttf файлы PT Sans Regular и Medium.

### Риск 2: Огромный PDF (сотни страниц, >50 МБ)

**Сценарий:** парк из 1500 устройств, все карточки развёрнуты.
**Защита:**
- Предупреждение в UI при генерации отчёта, если устройств > max_devices_in_pdf_warning.
- Опция «компактный PDF» — карточки устройств заменяются на строки таблицы с минимумом данных.

### Риск 3: Plotly не загружается offline

**Сценарий:** HTML-отчёт открывают там, где нет интернета, Plotly из CDN не подгружается, графики не показываются.
**Защита:** Plotly.js embed — инлайнится в HTML (размер ~3 МБ). Для экономии — опция `embed_plotly_offline: true` в конфиге.

### Риск 4: XSS через данные

**Сценарий:** в executive_summary от агента появился `<script>` или в описании устройства — HTML-теги.
**Защита:** Jinja2 по умолчанию экранирует HTML (`autoescape=True`). Безопасный вывод через `| safe` применяется только к доверенным полям (например, pre-rendered SVG графиков).

### Риск 5: Утечка PII в отчёте

**Сценарий:** в factor-store попали имена сотрудников, в отчёт — их локации в формате «Кабинет Иванова».
**Защита:** на уровне ingestion (трек E) — PII-фильтр. В отчёте — рендер как есть, но с опцией анонимизации в конфиге.

### Риск 6: Расхождение между HTML и PDF

**Сценарий:** HTML выглядит ок, в PDF что-то съехало из-за разной обработки CSS.
**Защита:** визуальные регрессионные тесты — сравнение PDF со screenshot-эталоном на фикстурах. См. TC-C-15.

### Риск 7: Устаревший executive_summary

**Сценарий:** пользователь изменил веса на странице 2, индексы пересчитались, но summary остался старым (он генерируется агентом и дорог в пересчёте).
**Защита:**
- Плашка «Сводка рассчитана при профиле X, текущий профиль — Y. [Обновить сводку]».
- Кнопка явного пересчёта summary.

### Риск 8: Снапшот конфигурации устарел

**Сценарий:** отчёт генерируется через 5 минут после расчёта, за это время успели сменить модель LLM.
**Защита:** CalculationSnapshot фиксируется **в момент расчёта**, не в момент генерации отчёта. Хранится в объекте Report.

### Риск 9: Конфликт зон индекса в отчёте и в весах

**Сценарий:** в weights.yaml `green_threshold: 75`, в report_config.yaml — `green_zone: 80`. Путаница.
**Защита:** пороги зон — **единый источник правды в weights.yaml**. report_config.yaml читает их оттуда, дублирование убрать. См. TC-C-21.

### Риск 10: Длинные таблицы ломают PDF-пагинацию

**Сценарий:** таблица устройств растягивается на несколько страниц, заголовок не повторяется.
**Защита:** `thead { display: table-header-group }`, `tr { page-break-inside: avoid }`. Проверяется тестом TC-C-18.

## Тест-кейсы

### Unit-тесты построения Report

**TC-C-1. ReportBuilder собирает корректный Report из списка HealthResult.**
- Вход: 3 HealthResult с известными индексами (20, 55, 90).
- Ожидание: Report.fleet_summary.zone_counts = {red:1, yellow:1, green:1}, average_index = 55.

**TC-C-2. Top-patterns корректно кластеризует массовые проблемы.**
- Вход: 4 устройства одной модели, у всех ошибка C6000.
- Ожидание: 1 pattern типа mass_issue с 4 affected_device_ids.

**TC-C-3. Top-patterns выявляет локационный паттерн.**
- Вход: 8 устройств в офисе "Ленинградка", у 6 — разные ошибки, но все с низким индексом.
- Ожидание: 1 pattern типа location_cluster.

**TC-C-4. Top-patterns ограничен 5 элементами.**
- Вход: 20 потенциальных паттернов.
- Ожидание: возвращается ровно 5, отсортированных по количеству затронутых устройств.

### Тесты HTML-рендеринга

**TC-C-5. HTML содержит все обязательные секции.**
- Ожидание: в HTML есть элементы с `id="section-summary"`, `id="section-patterns"`, `id="section-distribution"`, `id="section-table"`, `id="section-devices"`, `id="section-config"`.

**TC-C-6. Интерактивные элементы присутствуют в HTML.**
- Ожидание: найдены `<details>` для карточек, `<input class="search-bar">`, JS-скрипт сортировки таблицы.

**TC-C-7. Якорные ссылки работают.**
- Сценарий: в таблице клик на "MFP-042".
- Ожидание: в DOM есть `href="#device-MFP-042"` и соответствующий `id="device-MFP-042"`.

**TC-C-8. Экранирование HTML.**
- Вход: device_id = `<script>alert(1)</script>`.
- Ожидание: в HTML экранированные `&lt;script&gt;`, без исполняемого JS.

**TC-C-9. Скрытые элементы в PDF-режиме.**
- Сценарий: render_pdf.
- Ожидание: `.search-bar` и `.filter-chips` отсутствуют в CSS output (display:none применено).

### Тесты PDF-рендеринга

**TC-C-10. PDF генерируется без ошибок.**
- Ожидание: `render_pdf(report)` возвращает bytes, начинающиеся с `%PDF-`.

**TC-C-11. PDF содержит ключевые заголовки.**
- Проверка через pypdf: extract_text() содержит "Отчёт о здоровье парка МФУ", имена секций.

**TC-C-12. PDF содержит все устройства.**
- Сценарий: report с 10 устройствами.
- Ожидание: text PDF упоминает все 10 device_id.

**TC-C-13. Количество страниц разумное.**
- Сценарий: 50 устройств.
- Ожидание: PDF содержит 15–30 страниц (не 1, не 200).

**TC-C-14. Шапка и номера страниц на месте.**
- Ожидание: на каждой странице PDF в header — дата отчёта, в footer — "N / M".

**TC-C-15. Визуальный регрессионный тест.**
- Сценарий: сравнение скриншота рендера с эталонным изображением (golden file).
- Ожидание: pixel-diff < 1%, при ухудшении — вручную ревизируем и обновляем эталон.

### Тесты обработки крайних случаев

**TC-C-16. Пустой отчёт (0 устройств).**

**TC-C-19. Все устройства в одной зоне.**
- Сценарий: все 50 — зелёная зона.
- Ожидание: распределение показывается с 1 столбцом, подпись «Все в зелёной зоне».

**TC-C-20. Нет модели ни у одного устройства.**
- Ожидание: категорийная разбивка — по локациям или confidence-зонам.

### Тесты конфигурации

**TC-C-21. Единый источник порогов зон.**
- Сценарий: в weights.yaml зелёная зона = 80, в report_config.yaml не указана.
- Ожидание: отчёт использует 80 из weights.yaml. Если в report_config переопределено — ошибка валидации «единый источник правды».

**TC-C-23. Конфиг auto-breakdown работает.**
- Сценарий: отчёт с 1 моделью, но несколькими локациями.
- Ожидание: CategoryBreakdown.use = "location".

### Тесты снапшота

**TC-C-24. Snapshot воспроизводит отчёт.**
- Сценарий: сохранён отчёт с snapshot. Через неделю — из того же snapshot + тех же данных собирается новый отчёт.
- Ожидание: индексы и зоны идентичны, executive_summary может отличаться (генерируется заново).

**TC-C-25. Snapshot содержит хэш файла.**
- Ожидание: Report.source_file_hash = SHA-256 от исходного файла.

### Accessibility-тесты

**TC-C-26. Заголовок страницы осмысленный.**

### Интеграционные тесты

**TC-C-29. Полный end-to-end: расчёт → отчёт.**
- Сценарий: загрузка fixture-файла, прогон агента, генерация HTML и PDF.

## Dependencies

- `Jinja2` (шаблонизатор)
- `WeasyPrint` (HTML → PDF)
- `plotly` (интерактивные графики в HTML)
- `matplotlib` или `altair` + `vl-convert-python` (статичные SVG для PDF)
- `pypdf` (валидация сгенерированных PDF в тестах)
- `Pillow` (сравнение скриншотов в визуальных тестах)

Системные зависимости (для Docker-образа):
- `libpango`, `libcairo`, `libgdk-pixbuf` — для WeasyPrint
- PT Sans Regular/Medium .ttf (или Inter) в `/usr/share/fonts/`
