# Инструкция для LLM: автопрогон lint и test

## Назначение этого документа

Этот файл — **исполняемая инструкция для LLM-ассистента в VS Code** (Cursor, Claude Code, Cline, Continue, Copilot Chat и др.). Пользователь прикрепляет этот файл к чату и даёт команду «пройди по этому файлу». LLM **самостоятельно выполняет команды, анализирует результаты и принимает решения**.

Задача — довести состояние проекта до «все линтеры и тесты зелёные» с минимальным участием пользователя.

---

## Как LLM должна использовать этот файл

**Прочитай файл целиком перед началом.** В нём — пошаговый сценарий и правила принятия решений.

**Выполняй шаги последовательно.** Не перепрыгивай. После каждого шага — докладывай о результате и ждать явного подтверждения от пользователя перед переходом к следующему.

**Используй встроенный terminal-tool** (или аналог в твоей среде) для запуска команд. Если такого инструмента нет — давай пользователю точные команды для копирования.

**Соблюдай принцип «не сломать работающее».** Если тест был зелёный, а стал красный после твоих правок — немедленно откати и разберись.

**Не трогай документы из `docs/track_*.md` и `architecture_overview.md`.** Это спецификация. Если код не соответствует документу — правь код, а не документ.

**Каждая правка — минимальная.** Один файл за раз. Одно исправление за раз. После каждого — перепрогон проверки.

## Фаза 2. Линтинг — приведение в порядок стиля и типов

**Цель:** добиться чистого вывода `make lint`.

### Шаг 2.1 — Первичный прогон

Выполни:

```bash
make lint 2>&1 | tee lint_output.log
```

`tee` сохранит вывод в файл для сравнения потом. Прочитай файл `lint_output.log`.

### Шаг 2.2 — Классификация ошибок

**НЕ ИСПРАВЛЯЙ НИЧЕГО ПОКА.** Сначала классифицируй.

Разбери вывод по категориям:

**A. Критичные (блокируют работу):**
- `F401` — import неиспользуемый (обычно безопасно, но может скрывать баг).
- `F821` — undefined name (реальный баг).
- `F811` — redefinition of unused name (баг).
- `E999` — syntax error (код не парсится).
- mypy ошибки типов в сигнатурах функций.

**B. Стилистические (автоматически исправляются):**
- `E501` — line too long.
- `E302` — expected 2 blank lines.
- `W291`, `W293` — trailing whitespace.
- `I001` — import block is un-sorted.
- `Q000` — single quotes вместо double.

**C. Предупреждения (требуют внимания):**
- `B008` — mutable default argument.
- `N802` — имя функции не соответствует convention.
- `ARG001` — неиспользуемый аргумент функции.

**D. mypy-ошибки:**
- Расхождения типов между модулями.
- Missing type annotations.

Сделай краткую сводку пользователю:

```
Линтер нашёл N проблем:
- Критичных: X
- Стилистических (автофикс): Y
- Предупреждений: Z
- mypy-ошибок: W

План действий:
1. Прогоню ruff --fix на стилистику (категория B) — безопасно.
2. После этого проанализируем оставшееся.
3. Критичные и mypy-ошибки — разберём вручную по одной.

Продолжать? (да / нет / покажи подробнее)
```

Жди ответа.

### Шаг 2.3 — Автоисправление безопасных ошибок

После подтверждения:

```bash
ruff check --fix .
ruff format .
```

Затем снова:

```bash
make lint 2>&1 | tee lint_output_after_autofix.log
```

**Сравни** количество ошибок до и после. Ожидаемое: снижение в разы (часто от сотен до десятков).

Доложи пользователю:

```
Автофикс применён.
Было: N ошибок.
Стало: M ошибок.
Осталось разобрать:
- Критичных: X
- Предупреждений: Y  
- mypy: Z
```

### Шаг 2.4 — Разбор критичных ошибок (категория A)

Для каждой критичной ошибки:

1. Прочитай соответствующий файл на 20 строк вокруг проблемы.
2. Определи корневую причину.
3. Если это баг — исправь. Если это ложное срабатывание или требует обсуждения — задокументируй в отчёте и пропусти.

**Правило безопасности:** если в одном файле нужно сделать больше 3 правок, остановись и спроси пользователя:

```
В файле X нужно несколько правок подряд. Чтобы не усложнять ревью,
предлагаю показать diff после каждой правки. Или вы доверяете
сделать их за раз?
```

После каждой правки:
```bash
ruff check <файл>
mypy <файл>
```

Только того файла, который тронул.

### Шаг 2.5 — Разбор mypy-ошибок

mypy — самая коварная категория. Ошибки часто указывают на **реальные контрактные проблемы между модулями**.

Для каждой mypy-ошибки:

1. Прочитай файл с ошибкой.
2. Прочитай файл, из которого импортируется проблемный тип.
3. **Проверь соответствие с `docs/architecture_overview.md`** раздел «Межмодульные контракты» — там описаны правильные интерфейсы.
4. Если код расходится с документом — правь **код**, не документ.
5. Если документ сам по себе расплывчатый — пометь для обсуждения, не трогай.

**НИКОГДА не используй `# type: ignore` без явного обоснования.** Если хочется — это сигнал, что есть реальная проблема.

### Шаг 2.6 — Разбор предупреждений (категория C)

Обычно это не срочно. Предложи пользователю:

```
Осталось N предупреждений (категория C). Они не блокируют работу,
но могут указывать на потенциальные проблемы. Варианты:

1. Разобрать сейчас (15-30 минут).
2. Создать issue в трекере и пойти дальше.
3. Добавить в .ruff.toml exceptions, если мы осознанно не хотим их чинить.

Что выбираете?
```

### Шаг 2.7 — Финальная проверка lint

```bash
make lint
```

**Цель:** вывод `All checks passed!` или аналогичный (0 ошибок).

Если не удалось достичь нуля — сделай отчёт:

```
Фаза 2 завершена частично.
Ошибок было: N.
Ошибок стало: M.
Осталось нерешённым: список с причинами, почему не починили.

Рекомендация: [продолжить в Фазу 3 / остановиться и разобраться].
```

## Фаза 3. Тесты — проверка корректности

**Цель:** добиться прохождения всех тестов, релевантных для текущего окружения.

### Шаг 3.1 — Разведка тестов

Выполни:

```bash
pytest --collect-only -q 2>&1 | tee tests_collected.log
```

Это покажет **список всех тестов без запуска**. Посчитай:
- Сколько всего тестов.
- По каким файлам распределены.
- Есть ли тесты с маркерами (`@pytest.mark.integration`, `@pytest.mark.e2e`).

Доложи:

```
Обнаружено тестов: N.
По файлам:
- tests/test_calculator.py — X тестов
- tests/test_agent.py — Y тестов
...

С маркерами:
- @pytest.mark.unit — A тестов (не требуют внешних сервисов)
- @pytest.mark.integration — B тестов (требуют Qdrant)
- @pytest.mark.e2e — C тестов (требуют всё + LLM)
- Без маркеров — D тестов
```

### Шаг 3.2 — Прогон только unit-тестов

Unit-тесты — самые быстрые и не требуют внешних зависимостей. Начинаем с них.

```bash
pytest -v -m "unit" 2>&1 | tee tests_unit.log
```

Если маркеров нет, используй фильтр по именам файлов:

```bash
pytest -v tests/test_calculator.py tests/test_models.py 2>&1 | tee tests_unit.log
```

Проанализируй вывод.

### Шаг 3.3 — Разбор упавших unit-тестов

Для каждого упавшего теста:

**Шаг А. Определи категорию проблемы:**

1. **AssertionError** — тест вернул не то, что ожидалось. Это самое частое. Возможны две причины:
   - Код неправильный, тест правильный → чини код.
   - Тест неправильный, код правильный → чини тест по соответствующему TC-X-N из `docs/track_*.md`.

2. **ImportError / ModuleNotFoundError** — что-то не импортируется. Чаще всего — рассинхронизация имён модулей. Исправь импорты.

3. **TypeError, AttributeError** — нарушен контракт. Проверь `architecture_overview.md` раздел «Межмодульные контракты».

4. **Timeout** — тест висит. Возможно, ждёт что-то, чего нет. Обычно в unit-тестах не должно быть.

5. **FixtureError** — fixture не подгрузился. Проверь `tests/conftest.py` и `tests/fixtures/`.

**Шаг Б. Найди соответствующий TC-кейс в документации.**

Если тест называется `test_tc_a_3_critical_today` — найди TC-A-3 в `docs/track_A_health_index_formula.md`, раздел «Тест-кейсы». Там написано, что именно должно проверяться.

Сравни:
- Вход теста соответствует описанию TC?
- Ожидание теста соответствует описанию TC?
- Реализация в коде соответствует описанию формулы?

**Шаг В. Применяй правило приоритета:**

Документ > тест > код.

То есть:
- Если код ≠ документу → правь код.
- Если тест ≠ документу → правь тест.
- Если и код, и тест оба ≠ документу одинаково → возможно, они верны, а документ нечёткий. **Остановись и спроси пользователя.**

**Шаг Г. Минимальная правка.**

Одна ошибка — одна правка — один перезапуск теста:

```bash
pytest tests/test_X.py::test_Y -v
```

Если позеленел — следующая ошибка.

### Шаг 3.4 — Регрессионная проверка после каждой правки

**КРИТИЧЕСКОЕ ПРАВИЛО.**

После каждых 3-5 правок тестов перепрогоняй **все** unit-тесты:

```bash
pytest -v -m "unit"
```

Если тест, который раньше был зелёным, стал красным — это **регрессия**, откати последнюю правку и разберись.

Способ определить регрессию:
```bash
# Сравнение с прошлым прогоном
diff tests_unit.log tests_unit_previous.log | head -50
```

Или по количеству passed:
- Было: `passed=20, failed=5`.
- Стало: `passed=18, failed=7`. Регрессия.
- Стало: `passed=22, failed=3`. Прогресс.

### Шаг 3.5 — Прогон интеграционных тестов

Только если доступен Qdrant (проверь: `curl http://localhost:6333/ 2>/dev/null`).

```bash
pytest -v -m "integration" 2>&1 | tee tests_integration.log
```

**Если Qdrant недоступен** — сообщи пользователю:

```
Integration-тесты требуют запущенного Qdrant.
Варианты:
1. Запустить: `make up` или `docker-compose up -d`.
2. Пропустить этот этап, вернуться позже.

Что делать?
```

Если запустили — разбирай так же, как unit-тесты.

### Шаг 3.6 — Прогон e2e-тестов

Требует Qdrant И доступной LLM.

**Проверь доступность LLM:**

```bash
# Замени на реальный endpoint из .env
curl -s -o /dev/null -w "%{http_code}" $LLM_BASE_URL/models
```

Ожидаемо: 200, 401 (если нужен ключ — это тоже ок, значит endpoint живой).

Если недоступен:

```
E2E-тесты требуют запущенной локальной LLM на [адрес].
LLM недоступна.

Варианты:
1. Запустить LLM-сервер (vLLM, Ollama и т.д.).
2. Пропустить e2e, завершить автопрогон на успешных unit+integration.

Что делать?
```

Если всё доступно:

```bash
pytest -v -m "e2e" 2>&1 | tee tests_e2e.log
```

E2E-тесты медленные (минуты). Не прерывай. Если какой-то висит >5 минут — сообщи пользователю, возможно, надо увеличить timeout.

### Шаг 3.7 — Финальная сводка тестов

```bash
pytest --tb=no -q 2>&1 | tee tests_final.log
```

Доложи:

```
Результаты финального прогона тестов:

Всего: N
✓ Прошли: A
✗ Упали: B
⊘ Пропущены: C (из-за недоступного окружения)

По категориям:
- Unit: X/Y прошли
- Integration: P/Q прошли
- E2E: M/N прошли

Оставшиеся красные тесты:
[список с причинами и рекомендациями]
```

## Правила безопасности — что LLM НЕ должна делать

### Категорически запрещено

**Не менять файлы спецификаций.** `docs/track_*.md`, `docs/architecture_overview.md`, `docs/vscode_llm_playbook.md` — это source of truth. Если нашла противоречие — сообщи пользователю, не правь сама.

**Не удалять тесты, чтобы пройти.** Удаление падающего теста — это обман. Допустимый временный workaround — `@pytest.mark.skip(reason="...")` с конкретной причиной. Но только после обсуждения с пользователем.

**Не использовать `# type: ignore` и `# noqa` без причины.** Это глушит проблему, не решая её. Допустимо только когда явно указана причина в комментарии и есть согласие пользователя.

**Не ломать работающее.** Если тест был зелёный до твоих правок и стал красный — откати последнюю правку немедленно, разберись.

**Не делать массовые правки без ревью.** Если ошибка одного типа встречается в 50 местах, не правь все 50 за раз. Покажи на одном примере, получи подтверждение от пользователя, потом применяй ко всем.

**Не устанавливать новые зависимости без разрешения.** Если тест требует библиотеку, которой нет в `pyproject.toml` — остановись, спроси пользователя.

**Не менять версии существующих зависимостей.** Даже если это «решает» проблему. Может сломать что-то ещё.

### Требует явного согласия пользователя

- Установка новых пакетов.
- Изменение конфигов (`.ruff.toml`, `pyproject.toml`, `mypy.ini`).
- Добавление `skip`-маркеров к тестам.
- Изменение Makefile.
- Любые правки в более чем 5 файлов подряд.
- Коммиты в git.

## Формат общения с пользователем

**Сообщения должны быть короткими и структурированными.**

После каждого шага:

```
Шаг X.Y завершён.
Результат: [краткий статус].
Нашёл: [что нашёл].
Планирую: [следующее действие].
Продолжать? (да / покажи подробнее / стоп)
```

**Длинные выводы команд — в кратком виде.** Не вставляй в чат 500 строк лога. Сохрани в файл, покажи 5-10 ключевых строк и общую статистику.

**Ошибки — со ссылкой на файл и строку.** «Ошибка в `tools/calculator.py:45` — функция `compute_R` делает X, а должна делать Y согласно track_A раздел 3».

**Прогресс — в числах.** «Было 123 ошибки, стало 15, осталось разобрать 15 mypy-замечаний».

## Примеры принятия решений в частых ситуациях

### Ситуация 1: Тест упал на AssertionError

**Тест:**
```python
def test_tc_a_3_critical_today():
    result = calculate_health_index([critical_factor], weights=default_weights)
    assert result.health_index == 10
```

**Ошибка:** `assert 40 == 10`.

**Действия LLM:**

1. Найти TC-A-3 в `docs/track_A_health_index_formula.md`:
   > «Critical сегодня с высоким фьюзером. Вход: 1 фактор, S=60, R=1, C=1.5, A=1. Ожидание: penalty=90, H=10.»

2. Проверить входные данные теста — соответствуют ли описанию TC?
   Если `critical_factor` имеет `C=1.0` вместо `C=1.5` — это баг в **фикстуре теста**.

3. Проверить код `calculate_health_index` в `tools/calculator.py`:
   Если формула даёт `100 - 60*1*1*1 = 40` — значит, код не учитывает `C=1.5`. Баг в **коде**.

4. Правка: один из двух вариантов выше, в зависимости от того, где расхождение.

### Ситуация 2: mypy ругается на несоответствие типов

**Ошибка:**
```
agent/core.py:156: error: Argument 1 to "execute" of "ToolRegistry" has incompatible type "dict[str, Any]"; expected "ToolCall"  [arg-type]
```

**Действия LLM:**

1. Открыть `agent/core.py` на строке 156. Посмотреть, что вызывается.
2. Открыть `agent/tools/registry.py`, посмотреть сигнатуру `execute`.
3. Проверить `docs/architecture_overview.md` раздел «Межмодульные контракты» — какой контракт у `ToolRegistry.execute`.
4. Если `execute` должен принимать `ToolCall`, но вызывающая сторона передаёт `dict` — нужно сконвертировать: `ToolCall(**tc_dict)` перед вызовом, или привести сигнатуру к `dict`, если документ это допускает.


## Чек-лист завершения

Автопрогон считается успешным, когда:

- [ ] `make lint` завершается без ошибок (exit code 0).
- [ ] `make test` завершается без упавших тестов (в доступном окружении).
- [ ] Не осталось `# type: ignore` или `@pytest.mark.skip` без явных причин в комментариях.
- [ ] Все изменения в коде соответствуют `docs/track_*.md`.
- [ ] Пользователю отправлен финальный отчёт.
- [ ] Все лог-файлы (`lint_output.log`, `tests_final.log` и т.д.) сохранены в рабочей директории для ревью.

## Если автопрогон завис или не получается

Если LLM упёрлась в проблему, которую не может решить:

1. Зафиксируй последнее стабильное состояние.
2. Сделай отчёт:

```
Автопрогон остановлен на Фазе X, Шаге Y.

Что успели:
- Lint: [статус]
- Tests: [статус]

На чём застряли:
[описание проблемы в 2-3 предложениях]

Что я пробовала:
1. [попытка 1] — результат.
2. [попытка 2] — результат.

Рекомендации:
1. [что стоит сделать человеку]
2. [альтернативный подход]

Продолжать или остановиться?
```

Это **нормальный** результат. Лучше сказать «я не могу решить эту задачу», чем намутить хуже.

## Важное последнее

Этот файл — не магия. Он не заменит понимания проекта. LLM может прогнать lint и test, но:

- Она не знает бизнес-контекста.
- Она может сделать правку, которая технически правильная, но ломает замысел.
- Она не видит всю картину, только те файлы, которые прикреплены.

**Пользователь должен ревьюить результаты.** Минимум — смотреть diff по коммитам перед push в репозиторий.

Автопрогон — это помощник, а не замена разработчика.
