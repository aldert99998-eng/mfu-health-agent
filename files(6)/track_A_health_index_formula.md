# Трек A — Формула расчёта индекса здоровья МФУ

## Назначение и границы ответственности

Этот трек описывает **детерминированный расчёт численного индекса здоровья** (1–100) для отдельного МФУ и показателя **уверенности в оценке** (confidence, 0.2–1.0).

### Что входит в трек

- Математическая модель индекса и confidence
- Шкалы и пороги (Severity, зоны индекса)
- Коэффициенты и модификаторы (Repeatability, Context, Age)
- Правила обработки крайних случаев (нет данных, нет модели, скачки индекса)
- Структура конфигурации `weights.yaml`

### Что НЕ входит в трек

- Извлечение факторов из данных (это трек E — Ingestion)
- Получение Severity из документации (это трек D — RAG)
- Визуализация индексов (это трек C — Report)
- Логика рассуждения агента (это трек B — Agent)

Расчётный модуль — **чистая функция**: получает на вход факторы и веса, возвращает число. Никаких LLM-вызовов, никаких внешних зависимостей. Это делает его быстрым, повторяемым и тестируемым.

---

## Контекст — что этот трек задаёт для остальных

- **Треку B (Agent):** задаёт структуру входа для инструмента `calculate_health_index` — массив факторов с полями `(S, R, C, A, label, source)`.
- **Треку C (Report):** определяет, какие поля должен содержать результат расчёта для разложения индекса в отчёте.
- **Треку D (RAG):** задаёт, что классификатор Severity должен возвращать одно из пяти значений уровня критичности.
- **Треку E (Ingestion):** определяет требования к нормализации — ресурсы должны быть в процентах 0–100, даты в `datetime`, коды ошибок в каноничном виде.

---

## Детальное описание компонентов

### 1. Формула индекса здоровья

```
H = max(1, 100 - Σᵢ (Sᵢ · Rᵢ · Cᵢ · Aᵢ))
```

где сумма идёт по всем «проблемам» (ошибкам и предупреждениям) устройства за окно анализа.

**Компоненты penalty на одну проблему:**

| Компонент | Смысл | Источник значения |
|---|---|---|
| S (Severity) | Базовая критичность ошибки | RAG-классификатор по сервисной документации |
| R (Repeatability) | Коэффициент повторяемости | Подсчёт по истории событий |
| C (Context) | Контекстный модификатор | Правила + состояние ресурсов |
| A (Age) | Коэффициент давности | Экспонента от возраста события |

### 2. Шкала Severity (S)

Пять уровней критичности. Значения в скобках — базовые штрафы по умолчанию.

| Уровень | Описание | S |
|---|---|---|
| Critical | Полная неработоспособность: отказ фьюзера, лазерного блока, контроллера | 60 |
| High | Частичная неработоспособность: отказ сканера, постоянные замятия | 20 |
| Medium | Ограничения функциональности: сбои финишера, дуплекса | 10 |
| Low | Предупреждения: низкий тонер, приближение к сервисному интервалу | 3 |
| Info | Информационные события: замена картриджа, перезагрузка | 0 |

**Источник значения для конкретной ошибки:** RAG-поиск по сервисной документации с привязкой к модели устройства. Если RAG не нашёл — default `Medium` с флагом снижения confidence.

**Правило одного Critical в день.** Если за один день у устройства несколько ошибок уровня Critical, в расчёт берётся **только одна с максимальным итоговым penalty**. Остальные игнорируются. Это предотвращает каскадное занижение индекса при одной физической поломке, которая проявляется несколькими связанными кодами.

### 3. Коэффициент повторяемости (R)

```
R = min(5, 1 + log₂(n))
```

где `n` — количество повторений **того же кода ошибки** за 14-дневное окно, включая текущий день.

| n повторений | R |
|---|---|
| 1 | 1.00 |
| 2 | 2.00 |
| 4 | 3.00 |
| 8 | 4.00 |
| 16+ | 5.00 (потолок) |

Логарифм выбран намеренно: разница между 1 и 3 повторениями должна быть ощутимой, между 8 и 16 — уже нет (устройство в любом случае плохо).

Потолок в 5 предотвращает обнуление индекса одной повторяющейся ошибкой.

### 4. Контекстный модификатор (C)

Учитывает **состояние ресурсов и пробега**. Применяется по правилам, привязанным к конкретному узлу.

| Условие | Применяется к ошибкам | Множитель |
|---|---|---|
| toner_level < 10 | Ошибкам печати, качества | 1.5 |
| drum_level > 90 | Ошибкам качества печати | 1.3 |
| fuser_level > 90 | Ошибкам закрепления, замятиям | 1.5 |
| mileage > service_interval | Любым ошибкам | 1.2 |
| Нет связи ошибки с ресурсом | — | 1.0 (no-op) |

**Связка «ошибка ↔ узел»** определяется агентом на основе данных, полученных через RAG (инструмент `classify_error_severity` возвращает поле `affected_components`).

Если применимо несколько правил — множители **перемножаются**, но итоговый C ограничен сверху:

```
C = min(1.5, ∏ applicable_modifiers)
```

Потолок `1.5` предотвращает доминирование контекста над самой ошибкой. Изначально обсуждался потолок `2.0` — был снижен после подъёма Critical до 60, чтобы `S × C` для Critical не превышало 90.

### 5. Коэффициент давности (A)

```
A = exp(-d / 14)
```

где `d` — число дней с момента события до даты расчёта.

| d, дней | A |
|---|---|
| 0 | 1.00 |
| 3 | 0.81 |
| 7 | 0.61 |
| 14 | 0.37 |
| 21 | 0.22 |
| 30 | 0.12 |

**Окно анализа — 30 дней.** Всё старше отбрасывается (A ≈ 0 при d > 45).

Параметр `τ = 14` (полураспад) подобран так, чтобы события 30-дневной давности давали ~12% вклада — были видны в тренде, но не доминировали.

### 6. Формула confidence

```
Conf = max(0.2, ∏ₖ cₖ)
```

где `cₖ` — штраф-множитель за каждую обнаруженную проблему с данными.

**Список штрафов по умолчанию:**

| Причина снижения | Множитель |
|---|---|
| RAG не нашёл код ошибки (Severity взят дефолтный) | 0.7 (на каждый такой случай) |
| Нет данных о ресурсах | 0.85 |
| Нет модели устройства в данных | 0.6 |
| Резкий скачок индекса (>30 пунктов от вчерашнего) | 0.8 |
| Аномальное количество событий (>3σ от медианы по парку) | 0.7 |
| Нет событий за окно и нет снимка ресурсов | 0.9 |

**Интерпретация зон confidence:**

| Зона | Диапазон | Индикация |
|---|---|---|
| Высокая | 0.85–1.00 | зелёная полоска |
| Средняя | 0.60–0.85 | жёлтая полоска |
| Низкая | 0.20–0.60 | красная полоска, пометка «предварительный» |

Минимум 0.2 гарантирует, что результат всегда имеет хоть какой-то вес — полный сброс до 0 означал бы «расчёт не проводился».

### 7. Обработка устройств без событий

**Три возможных режима** (выбирается на странице весов админ-панели):

| Режим | Поведение |
|---|---|
| `optimistic` | H = 100, Conf = 1.0 («нет новостей — хорошие новости») |
| `data_quality` | H = 100, Conf × 0.9 (честно отмечаем «данных мало») |
| `carry_forward` | Использовать последний известный H из истории |

Режим по умолчанию — `data_quality`. Для демо, где истории может не быть, `carry_forward` деградирует до `data_quality`.

### 8. Пороги зон индекса

По умолчанию:

| Зона | Диапазон |
|---|---|
| Зелёная | H ≥ 75 |
| Жёлтая | 40 ≤ H < 75 |
| Красная | H < 40 |

Пороги настраиваются через UI и сохраняются в профиле весов.

## Интерфейсы модулей

### Вход: структура factor

```python
@dataclass
class Factor:
    error_code: str              # "C6000"
    severity_level: str          # "Critical" | "High" | "Medium" | "Low" | "Info"
    S: float                     # 60 — значение Severity из весов
    n_repetitions: int           # 3 — количество повторов за 14 дней
    R: float                     # 2.58 — рассчитанное значение
    C: float                     # 1.5 — рассчитанное значение
    A: float                     # 1.0 — рассчитанное значение
    event_timestamp: datetime    # для расчёта A
    age_days: int                # d
    applicable_modifiers: list[str]   # ["fuser>90"] — какие правила C применились
    source: str                  # "Service Manual TASKalfa 3253ci, раздел 7.2"
    confidence_flags: list[str]  # ["rag_not_found"] — флаги для снижения confidence
```

### Вход: структура ConfidenceFactors

```python
@dataclass
class ConfidenceFactors:
    rag_missing_count: int                  # сколько ошибок без RAG-хита
    missing_resources: bool
    missing_model: bool
    abnormal_daily_jump: bool               # индекс упал >30 относительно вчерашнего
    anomalous_event_count: bool             # >3σ от медианы
    no_events_and_no_resources: bool
```

### Вход: структура WeightsProfile

Соответствует YAML-файлу, см. ниже раздел «Конфигурация».

### Выход: структура HealthResult

```python
@dataclass
class HealthResult:
    device_id: str
    health_index: int                       # 1..100
    confidence: float                       # 0.2..1.0
    zone: str                               # "green" | "yellow" | "red"
    confidence_zone: str                    # "high" | "medium" | "low"
    factor_contributions: list[FactorContribution]
    confidence_reasons: list[str]           # что снизило confidence
    calculation_snapshot: dict              # профиль весов на момент расчёта
    calculated_at: datetime
    reflection_notes: str | None            # из self-check, опционально

@dataclass
class FactorContribution:
    label: str                              # "Отказ фьюзера (C6000)"
    penalty: float                          # вклад в снижение индекса
    S: float
    R: float
    C: float
    A: float
    source: str
```

## Конфигурационные параметры

Все параметры формулы живут в `configs/weights/default.yaml`. Пользовательские профили — в `configs/weights/profiles/*.yaml`.

```yaml
profile_name: "default"
version: "1.0"

severity:
  critical: 60
  high: 20
  medium: 10
  low: 3
  info: 0

repeatability:
  base: 2                       # основание логарифма
  max_value: 5                  # потолок R
  window_days: 14

context:
  modifiers:
    toner_low:                  # toner < toner_low_threshold
      threshold: 10
      multiplier: 1.5
      applies_to_components: [print, quality]
    drum_high:
      threshold: 90
      multiplier: 1.3
      applies_to_components: [quality]
    fuser_high:
      threshold: 90
      multiplier: 1.5
      applies_to_components: [fusing, jam]
    mileage_over_interval:
      multiplier: 1.2
      applies_to_components: ["*"]
  max_value: 1.5                # потолок C

age:
  tau_days: 14                  # полураспад
  window_days: 30               # окно анализа

critical_per_day_limit: 1       # только одна Critical в день учитывается

confidence:
  min_value: 0.2
  penalties:
    rag_not_found: 0.7          # применяется на каждое вхождение
    missing_resources: 0.85
    missing_model: 0.6
    abnormal_daily_jump: 0.8
    anomalous_event_count: 0.7
    no_events_and_no_resources: 0.9

zones:
  green_threshold: 75
  red_threshold: 40

silent_device_mode: "data_quality"   # optimistic | data_quality | carry_forward
```

**Правила работы с конфигом:**
- Все параметры валидируются через Pydantic при загрузке.
- Некорректные значения (отрицательные, вне диапазонов) — отказ загрузки профиля с внятной ошибкой.
- Смена профиля не требует перезапуска приложения.
- Каждый профиль при применении логируется с именем и timestamp.

## Риски и крайние случаи

### Риск 1: индекс выходит за диапазон 1–100

**Сценарий:** сумма penalty может быть очень большой (например, 5 Critical × C=1.5 = 450).
**Решение:** `max(1, ...)` в формуле. Минимум — 1, не 0, чтобы индекс всегда был валидным числом.

### Риск 2: деление на ноль или логарифм от нуля

**Сценарий:** n_repetitions может быть 0, если событие ещё не произошло — но это невозможно, потому что мы считаем R только для реально существующих событий.
**Защита:** Pydantic-валидация `n_repetitions >= 1` на входе в модуль. Тестом проверяется.

### Риск 3: некорректные типы данных

**Сценарий:** в factor пришёл `S=None` (например, из-за бага в RAG).
**Решение:** модуль возвращает ошибку, не пытается угадать. Вышестоящая логика должна решить — пропустить фактор или перезапустить расчёт.

### Риск 4: confidence падает до нуля

**Сценарий:** много штрафов подряд дают произведение, близкое к 0.
**Защита:** `max(0.2, ...)`. Нижний порог 0.2 означает «результат крайне ненадёжен, но формально рассчитан».

### Риск 5: расчёт не согласован между перерисовками UI

**Сценарий:** пользователь крутит слайдеры весов, индексы прыгают по частям, потому что factor-store обновляется лениво.
**Решение:** пересчёт — атомарный для одного устройства. factor-store создаётся один раз после ingestion и иммутабелен. Меняются только веса, формула заново применяется ко всему factor-store за один проход.

### Риск 6: флуктуации индекса на границе зон

**Сценарий:** устройство в одном расчёте имеет H=74 (жёлтая), в следующем — H=76 (зелёная), при незначительных изменениях.
**Решение:** не решается в формуле, **решается в отчёте** (трек C). В UI — гистерезис: переход из зоны в зону показывается только при изменении ≥ 3 пунктов.

### Риск 7: калибровка весов под «красивое распределение»

**Сценарий:** аналитик крутит слайдеры так, чтобы распределение по зонам выглядело, как ему нравится, а не отражало реальность.
**Защита:** не техническая, а процессная — A/B сравнение двух профилей на странице весов, история изменений, привязка профиля к отчёту (snapshot).

## Тест-кейсы

### Unit-тесты формулы (чистая математика)

**TC-A-1. Здоровое устройство.**
- Вход: пустой список факторов.
- Ожидание: H = 100, Conf = 1.0, zone = "green".

**TC-A-2. Единичная ошибка Medium.**
- Вход: 1 фактор, S=10, R=1, C=1, A=1.
- Ожидание: penalty = 10, H = 90, zone = "green".

**TC-A-3. Critical сегодня с высоким фьюзером.**
- Вход: 1 фактор, S=60, R=1, C=1.5, A=1.
- Ожидание: penalty = 90, H = 10, zone = "red".

**TC-A-4. Потолок R.**
- Вход: 1 фактор с n=100 (должен дать R > 5 без потолка).
- Ожидание: R = 5 (потолок сработал).

**TC-A-5. Потолок C.**
- Вход: 1 фактор, применяются все модификаторы C одновременно (toner_low × fuser_high = 1.5 × 1.5 = 2.25).
- Ожидание: C = 1.5 (потолок сработал).

**TC-A-6. Старые события затухают.**
- Вход: 1 фактор, S=60, R=1, C=1, A рассчитан для d=30.
- Ожидание: A ≈ 0.12, penalty ≈ 7, H ≈ 93.

**TC-A-7. Несколько Critical в один день.**
- Вход: 3 фактора уровня Critical с одинаковой датой.
- Ожидание: в расчёт попадает только 1 (с максимальным penalty), остальные 2 отмечены в результате как `suppressed_by_daily_limit`.

**TC-A-8. Минимум индекса.**
- Вход: 10 факторов с S=60, C=1.5 (сумма penalty ≈ 900).
- Ожидание: H = 1 (не 0, не отрицательное).

**TC-A-9. Минимум confidence.**
- Вход: 20 ошибок без RAG (0.7^20 ≈ 0.0008).
- Ожидание: Conf = 0.2 (нижний потолок).

### Unit-тесты правил применения Context

**TC-A-10. Модификатор fuser_high применяется только к ошибкам замятия/закрепления.**
- Вход: 2 фактора — один "ошибка сканера" (не должен), один "замятие" (должен), fuser_level=95.
- Ожидание: первый C=1.0, второй C=1.5.

**TC-A-11. Несколько применимых модификаторов перемножаются до потолка.**
- Вход: фактор «ошибка печати» с toner=5 и drum=95.
- Ожидание: C = min(1.5, 1.5 × 1.3) = 1.5.

### Интеграционные тесты с реальным factor-store

**TC-A-12. Пересчёт при смене профиля.**
- Сценарий: в factor-store записаны факторы для 10 устройств. Меняется профиль с дефолтного на «строгий» (Critical=80).
- Ожидание: индексы пересчитаны мгновенно (без обращений к LLM/RAG), распределение по зонам изменилось.

**TC-A-13. Разные устройства с одинаковыми ошибками дают разные индексы из-за контекста.**
- Сценарий: два устройства с одной и той же ошибкой C6000 (Critical). У первого fuser_level=40 (норма), у второго 95 (критично).
- Ожидание: H у второго ниже.

### Тесты обработки крайних случаев

**TC-A-14. Устройство без событий в режиме `data_quality`.**
- Вход: factor_store пустой для устройства, silent_device_mode=data_quality.
- Ожидание: H=100, Conf=0.9.

**TC-A-15. Устройство без событий в режиме `optimistic`.**
- Ожидание: H=100, Conf=1.0.

**TC-A-16. Отсутствует модель устройства.**
- Вход: фактор с `source=None` из-за отсутствия модели.
- Ожидание: confidence ×0.6, в `confidence_reasons` — "модель не определена".

**TC-A-17. RAG не нашёл 3 кода ошибок.**
- Ожидание: confidence × 0.7³ ≈ × 0.343.

### Регрессионные тесты

**TC-A-18. Снапшот весов воспроизводит индекс.**
- Сценарий: сохранён отчёт с индексом H=40 и snapshot профиля. Через месяц — применяем тот же snapshot к тем же входным данным.
- Ожидание: H=40 точь-в-точь.

**TC-A-19. Валидация конфига.**
- Сценарий: загрузка `weights.yaml` с `severity.critical: -10`.
- Ожидание: отказ загрузки с ошибкой «Severity values must be non-negative».

**TC-A-20. Иммутабельность factor-store.**
- Сценарий: изменение весов не меняет содержимое factor-store.
- Ожидание: factor-store остаётся идентичным до и после пересчёта.

## Dependencies

- Pydantic (валидация схем входа и конфига)
- PyYAML (загрузка профилей)
- Стандартная библиотека Python: `math`, `datetime`, `dataclasses`

LLM, RAG, Qdrant — **не используются** в этом треке. Это гарантия детерминированности и скорости.
