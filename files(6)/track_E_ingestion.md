# Трек E — Универсальный ingestion входных файлов

## Назначение и границы ответственности

Этот трек описывает **подсистему приёма данных мониторинга**: получение произвольного файла с данными о МФУ (CSV, JSON, Excel), автоматическое определение структуры, семантический маппинг полей к внутренней модели, нормализацию и валидацию.

### Что входит в трек

- Детекция формата файла и парсинг (CSV, JSON, JSONL, XLSX)
- Уплощение вложенных структур
- Трёхступенчатый маппинг полей: синонимы → содержимое → LLM
- Профили маппинга (запоминание структуры источника)
- Нормализация значений (timestamp, коды, ресурсы)
- Валидация через Pydantic
- Разделение данных на события и снимки ресурсов
- UI-wizard загрузки файла

### Что НЕ входит в трек

- Расчёт индексов (трек A)
- Структура RAG (трек D) — хотя нормализованные имена моделей используются там
- Логика агента (трек B) — агент обращается к factor-store через tools
- Визуализация отчёта (трек C)

Ingestion работает **до** агента. После завершения ingestion агент получает готовый factor-store и работает с ним.

---

## Контекст — что этот трек задаёт для остальных

- **Для трека A:** определяет, какие поля гарантированно есть в `NormalizedEvent` (timestamp, error_code опциональный) и `ResourceSnapshot` (toner/drum/fuser/mileage в процентах).
- **Для трека B:** factor-store, к которому агент обращается через `get_device_events`, `get_device_resources`, `count_error_repetitions`. Контракт этих tools зависит от структуры, которую заполняет ingestion.
- **Для трека D:** имя модели устройства приходит в RAG как payload-фильтр. Нормализация моделей через словарь алиасов — общий для ingestion и RAG.
- **Для трека C:** поля `model`, `location`, `location_cluster` в отчёте приходят из ingestion. Если поле не заполнено — отчёт должен уметь это обработать (трек C, раздел «Обработка крайних случаев»).

---

## Ключевой принцип проектирования

**Стандартизированной схемы входного файла не существует.** Система должна принимать любой разумный формат, сама определять, где какие поля, и приводить их к единому внутреннему представлению.

Это означает:
- Схема — **вывод системы**, а не вход.
- Пользователь **подтверждает**, но не **описывает** структуру своих данных.
- Повторные загрузки того же формата работают автоматически без участия пользователя (через профили).

Противоположный подход (жёсткая схема с документацией) требует от пользователя «подогнать» свои выгрузки под формат — это высокий барьер входа и источник ошибок. Универсальный ingestion снимает этот барьер.

### 2. Уровень 1 — Детекция формата и парсинг

**Определение типа файла:**

| Признак | Приоритет | Что даёт |
|---|---|---|
| Расширение файла | 1 (первичный) | Начальное предположение |
| Magic bytes | 2 (проверка) | Защита от неправильных расширений |
| Первые байты текста | 3 (уточнение) | Различает JSON/CSV при совпадающих расширениях |

Пример логики: `.json` с первым байтом `[` или `{` → JSON. `.json` с первой строкой, похожей на JSON-объект, а далее ещё одна — JSONL. `.csv` с `;` или `,` в первой строке — CSV с соответствующим разделителем.

**Для CSV/TSV:**

- **Кодировка** — через `chardet` с анализом первых 64 КБ файла. Русские CSV часто в Windows-1251, реже — CP866.
- **Разделитель** — через `csv.Sniffer` на первых 10 КБ. Fallback: тестовый парсинг с `,`, `;`, `\t` — выбирается тот, что даёт наибольшее одинаковое число колонок во всех строках.
- **Заголовок** — эвристика: первая строка состоит из строковых значений без чисел → header. Если все строки однотипные (включая первую) — header отсутствует, генерируются `col_1, col_2, ...`.

**Для JSON:**

Три возможные структуры, автоопределяемые:

```json
[{"device": "A"}, {"device": "B"}]                   // Массив объектов
```

```json
{"events": [{"device": "A"}, {"device": "B"}]}       // Обёрнутый массив
```

```jsonl
{"device": "A"}
{"device": "B"}                                       // JSONL
```

Алгоритм:
1. Пробуем `json.load()` как единый JSON.
2. Если успех и корень — массив → используем напрямую.
3. Если успех и корень — объект → ищем в нём массивоподобные ключи, выбираем самый длинный (по числу элементов).
4. Если `json.load()` упал — пробуем по строкам как JSONL.

**Вложенные структуры.**

Реальные системы мониторинга часто выдают вложенные объекты:

```json
{
  "device": {"id": "MFP-042", "model": "Kyocera 3253"},
  "event": {"code": "C6000", "timestamp": "2026-04-18T10:00:00Z"},
  "resources": {"toner": 45, "drum": 78, "fuser": 92}
}
```

Такое уплощается автоматически через `pandas.json_normalize`:
- `device.id`
- `device.model`
- `event.code`
- `event.timestamp`
- `resources.toner`
- `resources.drum`
- `resources.fuser`

Далее уровень 2 работает с плоской структурой — имена полей с точками нормально воспринимаются синонимайзером.

**Для XLSX:**

- Используется `openpyxl` в read-only режиме.
- По умолчанию — первый лист. Если листов несколько, в UI показывается выбор.
- Формулы вычисляются на стороне Excel перед загрузкой (читаются как значения).
- Даты из Excel-serial автоматически приводятся к `datetime`.

**Результат уровня 1:**

```python
@dataclass
class ParsedData:
    df: DataFrame                      # плоская таблица
    format: FileFormat                 # enum
    encoding: str | None               # для CSV
    delimiter: str | None              # для CSV
    total_rows: int
    total_cols: int
    source_path: Path
    source_hash: str                   # SHA-256
    warnings: list[str]                # эвристические проблемы
```

### 3. Уровень 2 — Семантический маппинг полей

**Целевая внутренняя модель:**

| Внутреннее поле | Тип | Обязательность | Описание |
|---|---|---|---|
| `device_id` | str | обязательное | Уникальный идентификатор МФУ |
| `timestamp` | datetime | обязательное | Время события |
| `error_code` | str | условно | Код ошибки (только для event-записей) |
| `error_description` | str | опционально | Текстовое описание |
| `model` | str | желательное | Модель устройства |
| `vendor` | str | опционально | Производитель |
| `location` | str | опционально | Локация/офис |
| `status` | str | опционально | Текущий статус устройства |
| `mileage` | int | опционально | Пробег в страницах |
| `toner_level` | int | опционально | Уровень тонера, % |
| `drum_level` | int | опционально | Ресурс барабана, % |
| `fuser_level` | int | опционально | Ресурс фьюзера, % |

**Условная обязательность:**
- Если в строке есть `error_code` → это событие, `timestamp` и `device_id` обязательны.
- Если `error_code` отсутствует, но есть хоть один ресурс → это снимок ресурсов, `timestamp` и `device_id` обязательны.
- Записи без `device_id` отбрасываются безусловно.

**Запись может быть сразу и событием, и снимком:** если есть и `error_code`, и `toner_level` — нормализатор разделяет её на два: событие и snapshot.

#### 3.2 Ступень B — Эвристики по содержимому

Если по имени не опознали — смотрим на **значения** в колонке.

Правила (применяются сверху вниз, первое совпадение — победитель):

| Правило | Условие | Вывод |
|---|---|---|
| R1 | Все значения парсятся как datetime | `timestamp` |
| R2 | Все значения парсятся как `^[CJEFS]\d{3,5}$` или `^SC\d{3}$` | `error_code` |
| R3 | Все значения в `[0, 100]` + имя содержит `toner` | `toner_level` |
| R4 | Все значения в `[0, 100]` + имя содержит `drum|барабан` | `drum_level` |
| R5 | Все значения в `[0, 100]` + имя содержит `fuser|фьюзер` | `fuser_level` |
| R6 | Целые числа > 1000, имя содержит `count|pages|pageCount` | `mileage` |
| R7 | Уникальность значений близка к 100%, длина строки 3–20 | `device_id` |
| R8 | Уникальных значений мало (< 50), длина 2–100, не number | `model` или `location` (требует LLM для различения) |

Правило R7 требует большого числа строк (минимум 20), иначе уникальность не показательна.

**Этот этап подхватывает колонки с нестандартными именами, но характерным содержимым.**

#### 3.3 Ступень C — LLM-классификатор

Для оставшихся неопознанных колонок — один LLM-вызов.

Промпт (файл `agent/prompts/field_mapping.md`):

```
Ты — помощник по маппингу полей. Тебе дан список колонок из
файла данных мониторинга МФУ и примеры значений.

Твоя задача — для каждой колонки определить, какому из
стандартных полей она соответствует.

Стандартные поля:

- device_id — уникальный идентификатор устройства
- timestamp — время события
- error_code — код ошибки (C6000, E200, SC001 и т.д.)
- error_description — текстовое описание ошибки
- model — модель устройства
- vendor — производитель
- location — локация/офис/здание
- status — статус устройства
- mileage — пробег (страниц напечатано)
- toner_level — уровень тонера в процентах
- drum_level — ресурс барабана в процентах
- fuser_level — ресурс фьюзера в процентах

Если колонка не соответствует ни одному из полей —
отметь как "_ignore".

Входные данные:

{columns_and_samples}

Верни JSON:
{
  "mapping": {
    "<название_колонки>": "<целевое_поле_или_ignore>",
    ...
  }
}

Не добавляй пояснений, только JSON.
```

Подстановка `{columns_and_samples}`:

```
Колонка: "Дата_опроса"
Примеры значений: "2026-04-18 10:00", "2026-04-18 11:30", "2026-04-17 09:00", "2026-04-17 14:20", "2026-04-16 08:00"

Колонка: "Префикс_каб"
Примеры значений: "Ленинг_5", "Ленинг_5", "Тверск_12", "Ленинг_5", "Тверск_12"

Колонка: "Кол_замятий"
Примеры значений: 0, 2, 1, 0, 3
```

Промпт и формат подобраны так, чтобы LLM могла обработать 10–20 колонок за один вызов. Ответ парсится через Pydantic-модель:

```python
class FieldMappingResponse(BaseModel):
    mapping: dict[str, str]
```

**Fallback:** если LLM недоступна или вернула невалидный JSON после 2 retries — неопознанные колонки остаются неопознанными, пользователь увидит их на шаге подтверждения и сам выберет маппинг.

#### 3.4 Ступень D — Подтверждение пользователем

Даже если все колонки опознаны автоматически, в Streamlit показывается таблица подтверждения.

**UI-макет (текстовое описание для реализации):**

```
┌─────────────────────────────────────────────────────────────┐
│ Файл обработан. Проверьте маппинг полей:                    │
│                                                             │
│ ┌─────────────────┬─────────────┬──────────────────┬──────┐ │
│ │ Колонка файла   │ Примеры     │ Поле             │ Увер.│ │
│ ├─────────────────┼─────────────┼──────────────────┼──────┤ │
│ │ ID_устройства   │ MFP-042, .. │ device_id    ▼  │ ✓ выс│ │
│ │ Дата_события    │ 2026-04-18..│ timestamp    ▼  │ ✓ выс│ │
│ │ Код             │ C6000, C6020│ error_code   ▼  │ ✓ сред│ │
│ │ Модель          │ TASKalfa..  │ model        ▼  │ ✓ выс│ │
│ │ Кол_замятий     │ 0, 2, 1, 0  │ ignore       ▼  │ ? LLM│ │
│ │ Префикс_каб     │ Ленинг_5..  │ location     ▼  │ ? LLM│ │
│ └─────────────────┴─────────────┴──────────────────┴──────┘ │
│                                                             │
│ Уверенность: высокая — синонимы, средняя — по содержимому,  │
│ LLM — спросили модель, неизвестно — не удалось определить.  │
│                                                             │
│ [ Сохранить как профиль: ______________ ]                   │
│                                                             │
│ [ ← Назад ]   [ Подтвердить и продолжить → ]                │
└─────────────────────────────────────────────────────────────┘
```

**Ключевые моменты интерфейса:**

1. Каждая колонка с **уровнем уверенности** маппинга — визуальный индикатор.
2. Выпадающий список целевых полей + опция `_ignore` + опция `_custom` (для полей, не входящих в стандартные — например, внутренний tag).
3. При изменении маппинга — live-валидация (если `device_id` назначен на две колонки, показывается предупреждение).
4. Кнопка «Сохранить как профиль» — поле ввода имени профиля. Профиль записывается в `configs/schema_profiles/<n>.yaml`.

**Запомненные маппинги (профили):**

```yaml
# configs/schema_profiles/kyocera_fleet_monitor_v1.yaml
profile_name: "Система мониторинга Kyocera Fleet v1"
signature_hash: "a3f7e2e8..."         # SHA-256 от нормализованных имён колонок
detected_on: "2026-04-18T10:00:00Z"
last_applied: "2026-04-20T15:30:00Z"
apply_count: 5                         # сколько раз использовался

column_mapping:
  "ID_устройства": device_id
  "Дата_события": timestamp
  "Код": error_code
  "Модель": model
  "Кол_замятий": _ignore
  "Префикс_каб": location

unit_hints:                            # опционально
  "Тонер_%": percent                   # уже в процентах
  "Drum_ratio": fraction               # долю нужно умножить на 100
```

**Применение профиля при новой загрузке:**

```python
def try_apply_profile(df: DataFrame) -> SchemaProfile | None:
    signature = compute_signature(df.columns)
    for profile in load_all_profiles():
        if profile.signature_hash == signature:
            return profile
    return None
```

Если профиль найден — маппинг применяется автоматически, wizard пропускается с плашкой «Использован профиль X» и кнопкой «Изменить маппинг». Если пользователь нажал «Изменить» — wizard открывается с предзаполненными полями.

**Signature hash** считается так:
1. Берутся имена колонок из `df.columns`.
2. Каждое имя нормализуется (как в ступени A).
3. Имена сортируются алфавитно.
4. Считается SHA-256 от JSON-массива.

Одинаковый набор колонок независимо от порядка = одинаковый hash.

### 4. Уровень 3 — Нормализация и валидация

#### 4.1 Парсинг timestamp

Используется `dateutil.parser.parse` с fallback на известные форматы:

```python
FALLBACK_FORMATS = [
    "%d.%m.%Y %H:%M:%S",
    "%d.%m.%Y %H:%M",
    "%d.%m.%Y",
    "%Y-%m-%d %H:%M:%S",
    "%Y-%m-%d",
    "%d/%m/%Y %H:%M",
    "%m/%d/%Y %H:%M",             # US format
    "%Y%m%d%H%M%S",               # компактный
]
```

**Unix timestamp** (целое число > 10^9) парсится отдельным правилом: `datetime.fromtimestamp(value, tz=timezone.utc)`.

**Часовые пояса:** если timestamp без TZ, применяется UTC по умолчанию. Флаг в конфиге `configs/ingestion.yaml`:

```yaml
timestamp:
  default_timezone: "UTC"                # или "Europe/Moscow"
  warn_on_naive: true
```

Если запись имеет битый timestamp — она попадает в `invalid_records` с указанием строки и исходного значения. Не пытаемся «догадаться».

#### 4.2 Нормализация error_code

Канонический вид: верхний регистр, без пробелов, дефисов, префиксов.

Трансформации:
- `"c6000"` → `"C6000"`
- `"C-6000"` → `"C6000"`
- `"Error C6000"` → `"C6000"`
- `"Err.C6000 (fuser)"` → `"C6000"`
- `"SC 345"` → `"SC345"`

Реализация — через regex-чейн:

```python
def normalize_error_code(raw: str) -> str | None:
    if not raw or not isinstance(raw, str):
        return None
    s = raw.upper().strip()
    # Удаление префиксов
    s = re.sub(r'^(ERROR|ERR\.?|КОД|ОШИБКА)\s*:?\s*', '', s)
    # Удаление пробелов/дефисов
    s = re.sub(r'[\s\-]+', '', s)
    # Удаление скобок и их содержимого
    s = re.sub(r'\(.*?\)', '', s)
    # Проверка финального формата
    if re.match(r'^[A-Z]{1,3}\d{3,5}$', s):
        return s
    return None  # не распознано
```

Если код не соответствует ни одному паттерну — запись помечается как «нераспознанный код», но не отбрасывается. Агент при работе с ней получит Severity = Medium по дефолту.

#### 4.3 Канонизация моделей

Через словарь `configs/model_aliases.yaml`:

```yaml
canonical_models:
  "Kyocera TASKalfa 3253ci":
    aliases:
      - "TASKalfa 3253ci"
      - "3253ci"
      - "kyocera 3253ci"
      - "Kyocera 3253"
      - "TASKalfa-3253ci"
    vendor: Kyocera
    family: "TASKalfa 3000 series"
  
  "HP LaserJet Pro M404n":
    aliases:
      - "LaserJet Pro M404n"
      - "M404n"
      - "HP M404n"
    vendor: HP
```

Функция нормализации ищет совпадение в алиасах (с нормализованным сравнением), возвращает каноническое имя. Если не нашла — оставляет как есть + предупреждение в логах.

**Важно:** то же самое словарное сопоставление применяется в треке D (RAG) при индексации документации. Это гарантирует, что поиск по `models=["Kyocera TASKalfa 3253ci"]` найдёт документы, где модель записана как `"3253ci"`.

#### 4.4 Автоопределение единиц ресурсов

Уровни тонера/барабана/фьюзера могут приходить в разных единицах. Автоопределение по диапазону значений в колонке:

| Диапазон | Интерпретация | Трансформация |
|---|---|---|
| `[0, 100]` | Процент | Без изменений |
| `[0, 1]` | Доля | Умножить на 100 |
| `> 100` и `< 10_000` | Абсолютные (страниц ресурса) | Оставить, флаг `unit_raw=True` |
| Остальное | Подозрительно | Warning в логах |

Если в колонке смесь значений (часть в `[0,1]`, часть в `[0,100]`) — выбор делается по большинству + предупреждение.

**unit_raw=True** означает, что нужна дополнительная обработка в формуле C (раздел 4 трека A). В демо эти значения игнорируются — контекстные правила не применяются.

**Override в UI:** пользователь может указать единицы вручную на шаге подтверждения маппинга.

#### 4.5 Валидация через Pydantic

Каждая нормализованная запись валидируется:

```python
class NormalizedEvent(BaseModel):
    model_config = ConfigDict(frozen=True)
    
    device_id: str = Field(min_length=1, max_length=100)
    timestamp: datetime
    error_code: str | None = Field(default=None, pattern=r'^[A-Z]{1,3}\d{3,5}$')
    error_description: str | None = None
    model: str | None = None
    vendor: str | None = None
    location: str | None = None
    status: str | None = None

class ResourceSnapshot(BaseModel):
    model_config = ConfigDict(frozen=True)
    
    device_id: str
    timestamp: datetime
    toner_level: int | None = Field(default=None, ge=0, le=100)
    drum_level: int | None = Field(default=None, ge=0, le=100)
    fuser_level: int | None = Field(default=None, ge=0, le=100)
    mileage: int | None = Field(default=None, ge=0)
    service_interval: int | None = None
    unit_raw: bool = False
```

**Invalid records** — список записей, не прошедших валидацию, с причиной:

```python
class InvalidRecord(BaseModel):
    row_number: int                    # строка в исходном файле (1-indexed)
    raw_data: dict                     # исходные значения
    reason: str                        # человекочитаемая причина
    field: str | None                  # конкретное поле, если известно
```

Invalid records **не отбрасываются молча** — показываются пользователю на итоговом экране: «15 из 1240 записей не прошли валидацию. Вот примеры ошибок: ...». Пользователь решает — продолжить без них или поправить исходный файл.

#### 4.6 Разделение на события и снимки ресурсов

После валидации у нас пакет `NormalizedRecord` — записей, которые могут быть:

- **Только событием** (есть `error_code`, нет ресурсов).
- **Только снимком** (нет `error_code`, есть хотя бы один ресурс).
- **И тем и другим** (и код, и ресурсы в одной строке).

Алгоритм разделения:

```python
def split_to_events_and_resources(
    records: list[NormalizedRecord]
) -> tuple[list[NormalizedEvent], dict[str, ResourceSnapshot]]:
    events = []
    resources_by_device: dict[str, list[ResourceSnapshot]] = defaultdict(list)
    
    for r in records:
        if r.error_code:
            events.append(NormalizedEvent.from_record(r))
        
        if r.has_any_resource():
            resources_by_device[r.device_id].append(
                ResourceSnapshot.from_record(r)
            )
    
    # Для каждого устройства берём последний snapshot по timestamp
    resources: dict[str, ResourceSnapshot] = {
        device_id: max(snaps, key=lambda s: s.timestamp)
        for device_id, snaps in resources_by_device.items()
    }
    
    return events, resources
```

#### 4.7 Заполнение factor-store

Финальный шаг ingestion:

```python
def fill_factor_store(
    events: list[NormalizedEvent],
    resources: dict[str, ResourceSnapshot],
    metadata: IngestionMetadata
) -> FactorStore:
    store = FactorStore()
    
    # Группировка событий по device_id
    events_by_device = group_by(events, key=lambda e: e.device_id)
    for device_id, device_events in events_by_device.items():
        store.add_events(device_id, device_events)
    
    # Снимки ресурсов
    for device_id, snapshot in resources.items():
        store.set_resources(device_id, snapshot)
    
    # Device metadata (модель, локация) — из последнего события
    for device_id in set(events_by_device) | set(resources):
        meta = extract_device_metadata(device_id, events, resources)
        store.set_device_metadata(device_id, meta)
    
    # Fleet-уровневая метаинформация
    store.set_fleet_meta(metadata)
    
    # Заморозка — после freeze store становится иммутабельным
    store.freeze()
    
    return store
```

### 5. UI-wizard загрузки файла

Streamlit-wizard на странице 1 админ-панели (или отдельная страница «Загрузка данных»).

**Шаг 1 — выбор файла.**
- Drag-and-drop area.
- Поддерживаемые форматы: CSV, TSV, JSON, JSONL, XLSX.
- Максимальный размер: 50 МБ (настраивается).
- Превью первых 5 строк в «сыром» виде.

**Шаг 2 — автоопределение маппинга.**
- Таблица колонок с маппингом и уровнем уверенности (макет в разделе 3.4).
- Индикатор «Найден профиль X — применяем» или «Профиль не найден, применяем эвристики».
- Время этого шага — < 3 секунд без LLM, 5–15 секунд с LLM.

**Шаг 3 — подтверждение и редактирование.**
- Пользователь может изменить маппинг.
- Поле «Сохранить как профиль» (опционально).

**Шаг 4 — валидация.**
- Прогресс-бар нормализации.
- Итоговая сводка: всего записей / валидных / битых.
- Если битые есть — expandable-блок с примерами (первые 20 строк).
- Кнопки «Продолжить (использовать только валидные)» и «Отменить и поправить файл».

**Шаг 5 — подтверждение запуска расчёта.**
- Сводка factor-store: сколько устройств, сколько событий, окно дат.
- Кнопка «Запустить анализ» → переход на страницу дашборда с прогрессом работы агента.

**Критические моменты UX:**

- Ошибки показываются **на том шаге, где произошли** (не все в конце).
- Возврат к предыдущему шагу **не теряет введённые данные** — состояние в `st.session_state`.
- При ошибке чтения файла — понятное сообщение с подсказкой («Проверьте, что файл сохранён в UTF-8, попробуйте открыть его в Excel и сохранить заново»).

### 6. Расширяемость

Архитектура рассчитана на то, что список форматов и синонимов будет расти.

**Добавление нового формата** (например, XML или проприетарный):

1. Создать парсер в `data_io/parsers/xml_parser.py` с интерфейсом:
   ```python
   class XMLParser(FileParser):
       def parse(self, file_path: Path) -> ParsedData: ...
   ```
2. Зарегистрировать в `FormatDetector`:
   ```python
   FORMAT_HANDLERS = {
       FileFormat.CSV: CSVParser,
       FileFormat.JSON: JSONParser,
       FileFormat.XML: XMLParser,   # новый
   }
   ```
3. Добавить детекцию по magic bytes.

Всё остальное (маппинг, нормализация) работает без изменений.

**Добавление нового поля** (например, `ip_address`):

1. Дописать в `NormalizedEvent` и `ResourceSnapshot`.
2. Добавить синонимы в `configs/field_synonyms.yaml`.
3. Обновить LLM-промпт в `field_mapping.md`.
4. При необходимости — добавить правило в ступень B.

### 7. Поведение при отсутствии данных

| Сценарий | Поведение |
|---|---|
| Файл пустой | `EmptyFileError`, wizard блокируется |
| Нет колонки с `device_id` | `CriticalMappingError`, wizard требует исправления |
| Нет колонки с `timestamp` | То же |
| Есть device_id и timestamp, но нет `error_code` ни в одной строке | Файл интерпретируется как snapshot-only (только ресурсы). Агент будет работать с пустыми events, индексы в режиме `data_quality` (Conf=0.9) |
| Есть события, но нет ни одной колонки с ресурсами | Агент получит пустые ResourceSnapshot, `missing_resources=true` — confidence × 0.85 |
| Нет колонки с моделью | Все device_metadata.model = None, `missing_model=true` — confidence × 0.6 |

## Риски и крайние случаи

### Риск 1: большой файл не помещается в память

**Сценарий:** CSV на 500МБ, `pandas.read_csv()` валит процесс.
**Защита:**
- Жёсткий лимит `max_size_mb: 50` в конфиге.
- При превышении — `FileTooLargeError` с подсказкой «разбейте файл на части».
- В будущем — chunked-режим с потоковой обработкой (не в демо).

### Риск 2: кодировка определена неверно

**Сценарий:** `chardet` вернул `utf-8`, но файл на самом деле cp1251 — русские символы ломаются.
**Защита:**
- Перебор fallback-кодировок при ошибке декодирования.
- Превью первой строки в UI — пользователь видит, читается ли текст.
- Ручной выбор кодировки в wizard'е.

### Риск 3: колонка-маппинг назначена дважды

**Сценарий:** `"ID"` и `"Серийный_номер"` обе помечены как `device_id`.
**Защита:**
- Live-валидация в UI: предупреждение «device_id назначен на 2 колонки, выбран будет первый».
- Приоритет: меньшее normalized-имя колонки (детерминировано).

### Риск 4: LLM-mapping возвращает несуществующие имена полей

**Сценарий:** LLM придумала `"серийник"` вместо `device_id`.
**Защита:**
- Pydantic-валидация ответа с enum допустимых значений.
- Retry с уточнённым промптом при невалидном ответе.
- Fallback на `_ignore` для неопознанных.

### Риск 5: профиль применился неправильно

**Сценарий:** два разных источника случайно имеют одинаковый набор колонок по именам, но разный формат значений (например, даты в разных форматах).
**Защита:**
- Signature учитывает только имена колонок — это осознанный компромисс.
- На экране wizard'а плашка «Применён профиль X» с кнопкой «Изменить» — пользователь может заметить неправильное применение и исправить.
- В логах — событие `profile_applied` с именем и датой.

### Риск 6: timestamp интерпретирован в неправильной TZ

**Сценарий:** данные в московском времени, но без TZ-суффикса. По дефолту применяется UTC → события сдвигаются на 3 часа.
**Защита:**
- Warning в логах при наивных timestamps.
- Настройка `default_timezone` в конфиге.
- Вывод примеров timestamp на экране подтверждения: пользователь видит «2026-04-18 10:00 UTC».

### Риск 7: Unit-detection ошибся

**Сценарий:** все значения тонера в колонке 0.45–0.98 (доля), но система определила как «почти всё в [0,100]», применила правило «процент».
**Защита:**
- Чёткие пороги: если `max < 1.5` → fraction; если `max < 101` → percent; иначе raw.
- Пользовательский override в UI.
- Предупреждение при подозрительных диапазонах.

### Риск 8: много битых записей, пользователь не замечает

**Сценарий:** из 10К записей 3К битых, пользователь жмёт «Продолжить», думая, что всё ок.
**Защита:**
- Плашка на шаге 4: если `invalid / total > 10%` — красное предупреждение «Много битых записей, уверены?».
- В сводке финального отчёта — упоминание количества битых записей.

### Риск 9: PII в исходных данных

**Сценарий:** в колонке `location` встречаются имена сотрудников («Кабинет Иванова»).
**Защита:**
- В демо — рендер как есть, без анонимизации.
- В прод — опция «Анонимизировать PII» на шаге подтверждения, с regex-фильтром.
- Предупреждение при обнаружении характерных паттернов (email, телефон, ФИО-like).

### Риск 10: дубликаты событий

**Сценарий:** одно и то же событие попало в файл дважды (разные выгрузки, неправильное объединение).
**Защита:**
- Дедупликация по ключу `(device_id, timestamp, error_code)`.
- Предупреждение в warnings: «Удалено N дубликатов».
- Опция «Keep duplicates» в настройках (по умолчанию — off).

### Риск 11: структура JSON вложена глубже ожидаемого

**Сценарий:** JSON с вложенностью 5+ уровней, `json_normalize` создаёт сотни колонок с длинными именами.
**Защита:**
- `max_depth: 10` в конфиге — защита от рекурсивных структур.
- Если `after-normalize columns > 50` — warning «Необычно много колонок, проверьте структуру».

### Риск 12: одна и та же модель в разном регистре

**Сценарий:** в файле модели «Kyocera 3253ci», «kyocera 3253ci», «KYOCERA 3253CI» — это один и тот же устройство, но после нормализации они могут дать три разных канонических имени.
**Защита:**
- Сравнение в словаре алиасов — case-insensitive.
- Warning при множественных вариантах одной модели в одном файле.

### Unit-тесты парсинга

## Примечание: расширение словарей

Словари синонимов и алиасов моделей — **живые документы**. По мере появления новых источников данных они будут пополняться.

**Процесс обновления:**
1. Пользователь загружает новый формат, ручной маппинг.
2. Если маппинг показывает паттерн (например, новые синонимы) — они могут быть предложены для добавления в глобальный словарь.
3. Администратор ревьюит предложения и мёрджит в `configs/field_synonyms.yaml`.

В демо этот процесс не автоматизирован — словари правятся вручную. В прод-версии стоит добавить UI «Предложения по расширению словарей».
