## Как пользоваться этим набором документов

Набор разделён на **отдельные MD-файлы по трекам архитектуры** — это сделано для того, чтобы:

1. **Не переполнять контекст LLM-сессии / ревью** — в любой момент открывается только один нужный файл.
2. **Работать параллельно** — разные тестировщики могут отвечать за разные треки.
3. **Привязать тесты к владельцам кода** — каждый трек имеет чёткого владельца по архитектуре.

Для общего обзора достаточно этого файла. Для прогона — открывается конкретный план.

## Приоритизация

| Приоритет | Критерий | Когда тестировать |
|---|---|---|
| **P0 (Critical)** | Блокирует запуск или искажает индекс здоровья | Каждый билд, перед любым релизом |
| **P1 (High)** | Серьёзно ухудшает UX или точность, но не блокирует | Каждый спринт |
| **P2 (Medium)** | Ухудшает качество на периферии | Перед мажорным релизом |
| **P3 (Low)** | Косметика, редкие сценарии | По расписанию регрессии |

Распределение P0 по файлам:
- 01_ingestion: загрузка файла, freeze(), корректность маппинга
- 02_calculator: формула, потолок 1.5, зоны, детерминизм
- 03_rag: точность retrieval, payload-индексы
- 04_agent: MAX_TOOL_CALLS, валидность JSON, reflection
- 05_reporting: рендер HTML+PDF без падений, полнота данных
- 07_e2e: полный цикл batch-режима от файла до отчёта
- 08_contracts: frozen=True, иммутабельность factor_store после freeze
- 09_nonfunctional: таймауты, OOM, безопасность RAG

## Соглашения по ID тест-кейсов

Формат: `TC-<трек>-<NNN>`

- `TC-E-NNN` — трек E (ingestion)
- `TC-A-NNN` — трек A (calculator)
- `TC-D-NNN` — трек D (RAG)
- `TC-B-NNN` — трек B (agent)
- `TC-C-NNN` — трек C (reporting)
- `TC-UI-NNN` — UI Streamlit
- `TC-E2E-NNN` — сквозные
- `TC-CNT-NNN` — контракты
- `TC-NF-NNN` — нефункциональные

Нумерация внутри трека сквозная через все разделы файла.

## Критерии приёмки системы в целом

Агент считается готовым к релизу, если:

1. **100% P0-кейсов** из всех планов пройдены (0 падений).
2. **≥ 95% P1-кейсов** пройдены.
3. **Все 10 рисков** из раздела 12 архитектуры имеют пройденный защитный тест.
4. **Контракты не нарушены** (все тесты из 08_contracts зелёные).
5. **E2E-сценарий** (07) отрабатывает на reference-датасете за время ≤ SLA.
6. **Покрытие unit-тестами ≥ 80%** на критичных модулях: calculator, agent/core, ingestion_pipeline, rag/search.
7. **Визуальный regression** отчёта зелёный (05).
8. **Безопасность RAG** пройдена (09): никаких утечек чужих коллекций, API-ключи не в логах.

## Глоссарий (сокращённый, см. раздел 14 архитектуры)

- **H** — health index, 1..100
- **Factor** — (S, R, C, A) — компонент формулы
- **Zone** — green (≥75) / yellow (40–74) / red (<40)
- **Trace** — запись цепочки шагов агента
- **Profile** — именованный набор параметров
- **Signature** — хэш колонок для автомэппинга
- **Reflection** — самопроверка агента
- **Tool strategy** — native / guided / react
