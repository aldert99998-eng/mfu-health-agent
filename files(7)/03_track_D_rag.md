# Тест-план 03 — Трек D: RAG (Qdrant + BGE-M3 + Hybrid search)

**Область:** `rag/` — qdrant_client, embeddings, reranker, search, ingestion, evaluation
**Владелец:** QA-инженер трека D
**Связанные документы:** `track_D_rag_base.md`, `architecture_overview.md` раздел 6.4, 7

---

## Цель

Проверить, что RAG:
1. **Корректно индексирует** документы сервисной документации (4 коллекции).
2. **Находит релевантные чанки** по запросам агента (hybrid search + rerank).
3. **Не смешивает** данные разных коллекций.
4. **Устойчив** к пустым, шумным и экзотическим запросам.
5. **Метрики качества** (precision/recall/nDCG) не падают ниже порогов на eval-датасете.

## Контракты

- **Вход поиска:** `query: str`, `collection: str`, `filters: dict`, `top_k: int`.
- **Выход:** `list[SearchResult]` (frozen dataclass, см. 6.4 архитектуры).
- **Коллекции:** service_manuals, internal_guides, error_codes_db, device_metadata (или как в track_D).

## Критерии приёмки трека D

- Все **P0** пройдены.
- Метрики на eval ≥ порогов (TC-D-051, 052).
- Нет утечек между коллекциями (TC-D-070).
- Словари-мосты синхронизированы (TC-D-120, 121, 122).
- Индексация и поиск выдерживают производственный объём (см. 09_nonfunctional.md).
