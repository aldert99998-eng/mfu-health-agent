# Тест-план 01 — Трек E: Ingestion

**Область:** `data_io/` — parsers, field_mapper, normalizer, factor_store, ingestion_pipeline
**Владелец:** QA-инженер трека E
**Связанные документы:** `track_E_ingestion.md`, `architecture_overview.md` раздел 4, 6.3

---

## Цель

Проверить, что произвольный табличный файл (CSV/JSON/XLSX) корректно превращается в нормализованный и иммутабельный `FactorStore`, готовый для работы агента. Ни одна строка не должна быть потеряна или искажена. Процесс загрузки устойчив к ошибкам формата, переключениям вкладок и обновлению страницы.

## Контракты

- **Вход:** файл пользователя + `schema_profile` (опц.) + `field_synonyms.yaml` + `model_aliases.yaml` + `error_code_patterns.yaml`.
- **Выход:** `FactorStore` после `freeze()` с `NormalizedEvent`, `ResourceSnapshot`, `DeviceMetadata` на каждое устройство.
- **Инварианты:** после `freeze()` store иммутабелен; все `error_code` прошли через regex-нормализацию; все `model` сведены к каноническим именам.

## 1. Парсинг форматов (parsers.py)

### TC-E-001. Парсинг корректного CSV с заголовками (P0)
**Шаги:** подать файл `valid_standard.csv` со стандартным набором колонок.
**Ожидание:** формат определён как `csv`; возвращён DataFrame со всеми строками; типы базовые (строки).

### TC-E-002. Парсинг XLSX с несколькими листами (P1)
**Шаги:** XLSX с листами `errors`, `resources`, `metadata`.
**Ожидание:** парсер читает все листы, сохраняет имя листа в метаданных; по умолчанию основной лист — первый, остальные доступны по имени.

### TC-E-003. Парсинг JSON — массив объектов (P0)
**Шаги:** JSON вида `[{...}, {...}]`.
**Ожидание:** каждый объект → строка; вложенные объекты сериализованы в строку либо разворачиваются при наличии правила.

### TC-E-004. Парсинг JSON — NDJSON (P1)
**Шаги:** файл, где каждая строка — отдельный JSON-объект.
**Ожидание:** корректно читается построчно.

### TC-E-005. Автоопределение формата по содержимому (P1)
**Шаги:** подать файл `data.txt`, содержащий CSV.
**Ожидание:** по magic/sniff определён формат CSV, не отклонён из-за расширения.

### TC-E-006. Отклонение бинарного файла с подменённым расширением (P0, безопасность)
**Шаги:** `fake.exe` переименованный в `fake.csv`.
**Ожидание:** отклонение по сигнатуре magic bytes, ошибка `InvalidFileFormatError`. На диск сервера не сохраняется.

### TC-E-007. Разделители CSV: `,`, `;`, `\t` (P1)
**Шаги:** три файла с разными разделителями.
**Ожидание:** все распознаны автоматически через csv sniffer.

### TC-E-008. Смешанные разделители внутри одного файла (P2)
**Шаги:** `mixed_separators.csv`.
**Ожидание:** ошибка парсинга с указанием первой проблемной строки.

### TC-E-009. Кодировки UTF-8, UTF-8 BOM, Windows-1251 (P1)
**Шаги:** три идентичных по содержанию файла в разных кодировках.
**Ожидание:** кодировка определяется через `chardet`/аналог. Кириллица во всех случаях читается без кракозябр.

### TC-E-010. Повреждённый CSV (незакрытые кавычки) (P1)
**Ожидание:** ошибка с номером строки; частичного импорта нет.

### TC-E-011. Пустой файл (0 байт) (P2)
**Ожидание:** ошибка валидации «Файл пустой», не вызывает падения pipeline.

### TC-E-012. Файл только с заголовками (P2)
**Ожидание:** warning «Нет строк данных»; factor_store пустой, но не в ошибочном состоянии.

### TC-E-013. Нарушение схемы: отсутствует колонка `device_id` (P0)
**Ожидание:** явная ошибка с указанием, какая колонка отсутствует и какие синонимы искались.

### TC-E-014. Дубликаты заголовков колонок (P2)
**Ожидание:** ошибка валидации; не должно быть тихого затирания данных.

### TC-E-015. Очень длинные строки (>10К символов в одной ячейке) (P2)
**Ожидание:** читаются без усечения; в логах предупреждение о необычно длинном значении.

## 2. Field Mapping — 3 ступени + UI

### TC-E-020. Ступень 1: точное совпадение по `field_synonyms.yaml` (P0)
**Шаги:** колонка `device_serial` → синоним для `device_id`.
**Ожидание:** маппинг сделан автоматически без LLM; в логах метка `matched_by: synonyms`.

### TC-E-021. Ступень 2: совпадение по профилю (signature match) (P1)
**Шаги:** набор колонок ранее сохранён как профиль `kyocera_v1`; загружен файл с тем же набором.
**Ожидание:** сработал signature match; выдаётся плашка «Применён профиль kyocera_v1, проверьте».

### TC-E-022. Ступень 3: LLM-маппинг для незнакомых колонок (P1)
**Шаги:** колонка `err_cd_device_local` без синонима и без профиля.
**Ожидание:** вызов LLM с промптом `field_mapping.md`; ответ сопоставлен с `error_code`; confidence в трассировке.

### TC-E-023. LLM вернула невалидный ответ (P1)
**Шаги:** замокать LLM, чтобы вернула текст вместо JSON.
**Ожидание:** retry с guided JSON; при повторном провале — fallback на «требуется ручной маппинг», пользователь видит UI для ручного сопоставления.

### TC-E-024. Ручной override маппинга через UI (P0)
**Шаги:** автомэппинг поставил `device_id` на не ту колонку; пользователь исправляет в UI.
**Ожидание:** override сохранён; при следующей загрузке того же формата профиль учтёт изменение (либо спросит «сохранить как новый профиль?»).

### TC-E-025. Неоднозначный маппинг (две колонки претендуют на `device_id`) (P1)
**Ожидание:** агент / пользователь уведомлён о неоднозначности; не тихий выбор.

### TC-E-026. Сохранение нового профиля (P1)
**Шаги:** после успешного маппинга нажать «Сохранить как профиль X».
**Ожидание:** YAML создан в `configs/schema_profiles/x.yaml`; signature корректно вычислен.

### TC-E-027. Ложное срабатывание профиля (риск 10 из архитектуры) (P0)
**Шаги:** два разных типа файлов имеют одинаковый набор колонок, но разную семантику.
**Ожидание:** плашка «Применён профиль X, проверьте» видна и блокирует автоматический proceed без подтверждения.

## 3. Нормализация (normalizer.py)

### TC-E-030. Нормализация `error_code` через `error_code_patterns.yaml` (P0)
**Шаги:** на вход коды в разных форматах: `"C-6000"`, `"c6000"`, `"C 6000"`, `"error C6000"`.
**Ожидание:** все → `C6000`.

### TC-E-031. Нормализация `model` через `model_aliases.yaml` (P0)
**Шаги:** `"3253ci"`, `"TASKalfa 3253ci"`, `"Kyocera 3253"` → `"Kyocera TASKalfa 3253ci"`.

### TC-E-032. Код не матчится ни одному регексу (P1)
**Ожидание:** код сохраняется как есть, с флагом `confidence_flags: ["unknown_error_code_format"]`.

### TC-E-033. Модель не матчится ни одному алиасу (P1)
**Ожидание:** модель как есть + флаг; агент при обращении в RAG получит низкий score, что и ожидается.

### TC-E-034. Валидация через Pydantic (`NormalizedEvent`) (P0)
**Ожидание:** строка с невалидным `timestamp` (например, `"32.13.2025"`) отклоняется; импорт продолжается с остальными, невалидная помечается в отчёте об ingestion.

### TC-E-035. Timezone handling (P1)
**Шаги:** timestamps в разных TZ (UTC, MSK, без TZ).
**Ожидание:** все приводятся к UTC (или к заданной в конфиге TZ); логика документирована.

### TC-E-036. Числовые поля (`mileage`, `toner_level`) с запятой/точкой (P2)
**Ожидание:** `"12,5"` и `"12.5"` → `12.5`; `"95%"` → `95.0`.

### TC-E-037. Пропуски в необязательных полях (P1)
**Ожидание:** обработка без ошибок; поле = None.

### TC-E-038. Пропуски в обязательных полях (`device_id`, `timestamp`) (P0)
**Ожидание:** строка отбрасывается, учитывается в отчёте ingestion (`rows_skipped`).

### TC-E-039. Аномально большие значения (mileage = 99_999_999_999) (P2)
**Ожидание:** warning, но не отклонение (может быть реальным для кумулятивного счётчика).

### TC-E-040. Отрицательные значения там, где не должно быть (toner_level = −5) (P2)
**Ожидание:** clamp в [0, 100] + warning.

## 4. FactorStore — контракт E ↔ B

### TC-E-050. `add_events()` сохраняет в правильный device_id (P0)
### TC-E-051. `set_resources()` заменяет предыдущий snapshot (P1)
### TC-E-052. `set_device_metadata()` идемпотентен (P2)
### TC-E-053. `list_devices()` возвращает без дубликатов (P0)

### TC-E-054. `freeze()` делает store иммутабельным (P0)
**Шаги:** после freeze() вызвать `add_events()`.
**Ожидание:** `FrozenError` или аналог; данные не изменились.

### TC-E-055. Множественные `freeze()` подряд (P2)
**Ожидание:** идемпотентно, не падает.

### TC-E-056. `get_events(device_id, window_days=30)` — фильтр по окну (P0)
**Шаги:** 5 событий: 3 за последние 30 дней, 2 старше.
**Ожидание:** возвращено 3.

### TC-E-057. `count_repetitions(device_id, error_code, window_days)` — точный счёт (P0)
**Шаги:** 4 повтора C6000 + 1 старый.
**Ожидание:** с window=30 → 4; с window=365 → 5.

### TC-E-058. Обращение к несуществующему `device_id` (P1)
**Ожидание:** возвращается пустой список или `None`, не падение.

### TC-E-059. Конкурентный доступ (чтение из тула + запись из pipeline) (P1)
**Примечание:** по архитектуре freeze() гарантирует отсутствие записей после — но проверить, что тулы не получают «полусобранное» состояние до freeze().

## 5. Ingestion Pipeline (координатор)

### TC-E-070. Успешный полный прогон (P0)
**Шаги:** корректный файл на 1000 устройств → parse → map → normalize → freeze.
**Ожидание:** factor_store содержит 1000 устройств; отчёт ingestion: `rows_processed`, `rows_skipped`, `warnings`.

### TC-E-071. Идемпотентность (повторная загрузка того же файла) (P1)
**Ожидание:** либо ошибка «уже загружено» по хешу файла, либо создаётся новая сессия с другим `ingestion_id`. Поведение задокументировано.

### TC-E-072. Частичный сбой на нормализации (P1)
**Шаги:** 100 строк валидные, 5 невалидные.
**Ожидание:** 100 в store, 5 в отчёте как skipped; нет rollback валидных данных.

### TC-E-073. Checkpoint-механизм (если реализован) (P2)
**Шаги:** прервать на середине → перезапуск.
**Ожидание:** возобновление с чекпоинта, без дублей.

### TC-E-074. Лог ingestion сохранён в `storage/ingestion_checkpoints/` (P2)
**Ожидание:** есть файл с метаданными запуска.

### TC-E-075. Проверка целостности данных через контрольную сумму (P0)
**Шаги:** посчитать COUNT, SUM(mileage), distinct(device_id) до и после.
**Ожидание:** все сходятся с файлом.

## 6. UI загрузки — стабильность процесса ⭐

Этот раздел проверяет требование: **процесс загрузки не должен прерываться** при переключении вкладок и обновлении страницы управления.

### TC-E-080. Загрузка не прерывается при переключении вкладок дашборда (P0)
**Шаги:**
1. Начать загрузку файла ~25 МБ на странице «Dashboard / Загрузка».
2. При прогрессе 20–30% переключиться на вкладку «2. Веса», затем «3. LLM/Чат», затем «4. RAG-админ».
3. Подождать 10 секунд на каждой.
4. Вернуться на Dashboard.

**Ожидание:**
- Процесс продолжает работу в фоне (т.к. ingestion в `@st.cache_resource` / background task).
- Прогресс-бар показывает актуальное значение.
- По завершении factor_store корректно сформирован.
- В Streamlit session_state есть результат, а не потерян.

### TC-E-081. Загрузка не прерывается при F5 / reload страницы (P0)
**Шаги:**
1. Запустить загрузку, дождаться 30–50%.
2. F5.

**Ожидание:**
- После reload UI отображает активное задание ingestion с актуальным прогрессом.
- factor_store (singleton) не потерян, т.к. держится в `@st.cache_resource`.
- Session-state UI перезаполняется из cache_resource.
- По завершении результат корректный, строки не задублированы.

### TC-E-082. Одновременная загрузка двух файлов в разных сессиях (P1)
**Ожидание:** не блокируют друг друга; каждая сессия видит свой ingestion.

### TC-E-083. Отмена загрузки пользователем (P1)
**Шаги:** нажать «Отмена» при 50%.
**Ожидание:** pipeline останавливается на ближайшей безопасной точке, частичных изменений не остаётся, factor_store либо пуст, либо откатан.

### TC-E-084. Закрытие вкладки браузера во время загрузки (P1)
**Ожидание:** по возвращении пользователь видит состояние задачи (завершена / отменена / продолжает). Не «подвешенный» спиннер.

### TC-E-085. Потеря сети (DevTools offline) во время загрузки (P1)
**Ожидание:** сам файл уже на сервере (загружен); ingestion продолжается на сервере; UI после восстановления подхватывает состояние.

### TC-E-086. Drag-and-drop загрузка (P2)
**Ожидание:** аналогично через кнопку.

### TC-E-087. Превышение лимита размера файла (P0)
**Шаги:** загрузить файл 2× `max_size_mb`.
**Ожидание:** отклонение на клиенте; если прошло клиент — отклонение сервером с понятным сообщением.

## 7. Нагрузочные и граничные

### TC-E-090. Файл 100 000 строк (P1)
**Ожидание:** ingestion укладывается в SLA (определить — например, ≤ 60 сек); память не более Х ГБ.

### TC-E-091. Файл 1 000 000 строк (P2)
**Ожидание:** либо успешный прогон, либо graceful отказ с рекомендацией разбить; не OOM-краш.

### TC-E-092. 10 000 уникальных device_id (P1)
**Ожидание:** factor_store корректно индексирует; доступ `get_events()` — O(1)/O(log n).

### TC-E-093. Файл с нестандартными символами (emoji, RTL) (P2)
**Ожидание:** сохраняются корректно в NormalizedEvent.

## 8. Проверка словарей-мостов

### TC-E-100. Расхождение между `model_aliases.yaml` (E) и тем же файлом в RAG (D) (P0)
**Шаги:** в ingestion алиас `Kyocera 3253ci` → `TASKalfa 3253ci`, но в payload RAG хранится `Kyocera TASKalfa 3253ci`.
**Ожидание:** unit-тест синхронизации словарей падает (один YAML — источник правды).

### TC-E-101. Новый код ошибки, не покрытый regex (P2)
**Ожидание:** флаг `unknown_error_code_format`; агент при классификации всё равно может работать через текст.

## Критерии приёмки трека E

- Все **P0** пройдены.
- Контрольная сумма после ingestion совпадает с файлом (TC-E-075).
- Загрузка не прерывается при переключении вкладок и reload (TC-E-080, TC-E-081).
- freeze() обеспечивает иммутабельность (TC-E-054).
- Словари-мосты синхронизированы с треком D (TC-E-100).
