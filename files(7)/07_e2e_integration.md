# Тест-план 07 — End-to-End сценарии

**Область:** сквозные цепочки: Ingestion (E) → Agent (B) → Calculator (A) → Report (C) + RAG (D) + UI
**Владелец:** QA-команда
**Связанные документы:** `architecture_overview.md` раздел 5

---

## Цель

Проверить, что **вся система** работает согласованно как единое целое:
1. Файл → factor_store → per-device HealthResult → Report (HTML + PDF).
2. Диалог в чате → ответ на основе реальных данных и RAG.
3. Регрессионные сценарии с фиксированными ожидаемыми результатами.

E2E-тесты — **вершина пирамиды тестирования**. Медленные, но дают уверенность в релизе.

## Подготовка

- Fixture-файлы в `tests/fixtures/`.
- Seed-данные для RAG (индексированы заранее).
- Mock LLM для детерминизма (базовый случай) + реальная LLM для smoke.
- Eval dataset для агента.

## 1. Batch-режим — полный путь

### TC-E2E-001. Happy path: 50 устройств → отчёт (P0)
**Шаги:**
1. Стартовать приложение.
2. Страница Dashboard → загрузить `fixtures/fleet_50_standard.csv`.
3. Запустить анализ.
4. Дождаться завершения.
5. Открыть отчёт в UI.
6. Скачать PDF.

**Ожидание:**
- Ingestion: все 50 устройств в factor_store.
- Agent: 50 HealthResult, у каждого корректный health_index, zone, factor_contributions.
- Report: fleet_summary правильный (counts по zone соответствуют сумме per-device).
- PDF скачан, открывается, содержит все секции.
- Время выполнения в SLA (≤ 10 минут на локальной LLM).

### TC-E2E-002. Reference-датасет: 20 устройств с эталонными результатами (P0)
**Шаги:** fixture-файл + ожидаемые HealthResult + ожидаемый executive_summary pattern.
**Ожидание (с mock LLM для детерминизма):**
- Каждый health_index совпадает с эталоном.
- Каждая zone совпадает.
- factor_contributions соответствуют эталону.
- Executive summary содержит ожидаемые ключевые слова.

### TC-E2E-003. Реальная LLM — smoke (P1)
**Ожидание:** прогон с реальной LLM не падает; результаты в разумных диапазонах.

### TC-E2E-004. Файл с ошибочным маппингом → ручной override → успех (P0)
**Шаги:**
1. Загрузить файл, где автомэппинг ошибочен.
2. В UI исправить маппинг.
3. Запустить анализ.
**Ожидание:** override учтён, правильные результаты.

### TC-E2E-005. Пустой парк (0 ошибок на всех устройствах) (P1)
**Ожидание:** все устройства в green, executive_summary позитивный.

### TC-E2E-006. Массовая проблема (30/50 устройств с C6000) (P0)
**Ожидание:**
- В top_patterns есть «C6000» как массовый паттерн.
- В executive_summary упоминается эта проблема.
- RAG подтянул документацию по C6000.
- Агент дал конкретную рекомендацию.

### TC-E2E-007. Устройство с flag_for_review → видно в отчёте (P0)
**Ожидание:** в карточке устройства баннер / индикатор; executive_summary упоминает необходимость ручной проверки.

### TC-E2E-008. Пересчёт с другим WeightsProfile (P0)
**Шаги:**
1. Прогон с default.
2. Смена на «strict» профиль.
3. Кнопка «пересчитать».
**Ожидание:**
- factor_store не пересчитывается (остался immutable после freeze).
- LLM не вызывается повторно для severity (кэш).
- Новые HealthResult отражают новые веса.
- Отчёт перегенерирован с новым calculation_snapshot.

## 2. Chat-режим — полный путь

### TC-E2E-020. Вопрос про конкретное устройство после batch (P0)
**Шаги:**
1. Прогон batch на 50 устройствах.
2. Страница 3 Chat: «Почему MFP-017 в красной зоне?»
**Ожидание:**
- Агент вызывает get_device_events, возможно classify_error_severity.
- Ответ ссылается на конкретные ошибки из файла.
- Ответ упоминает health_index и факторы снижения.
- Trace раскрывается, видны шаги.

### TC-E2E-021. Вопрос про парк (статистика) (P0)
**Шаги:** «Сколько HP устройств в красной зоне?»
**Ожидание:** корректное число; вызов `get_fleet_statistics`.

### TC-E2E-022. Вопрос про техническую проблему (RAG) (P0)
**Шаги:** «Как исправить ошибку C6000 на TASKalfa 3253ci?»
**Ожидание:** вызов `search_service_docs`; ответ содержит процедуру с source_refs.

### TC-E2E-023. Многошаговый диалог (P1)
**Шаги:**
1. «Покажи худшие 5 устройств».
2. «Что общего у первых двух?»
3. «Дай рекомендацию по исправлению».
**Ожидание:** контекст сохраняется, агент понимает ссылки между вопросами.

### TC-E2E-024. Вопрос, на который агент не может ответить (данных нет) (P1)
**Ожидание:** честный ответ «данных недостаточно», без галлюцинаций.

### TC-E2E-025. Параллельный chat во время batch (P1)
**Ожидание:** пользователь может задавать вопросы; агент отвечает на основе уже обработанных устройств.

## 3. Стабильность UI + backend

### TC-E2E-030. Переключение вкладок во время batch не прерывает (P0)
**Шаги:**
1. Batch на 100 устройств.
2. При 20% — на страницу 2 Weights, изменить слайдер, не сохранять.
3. На страницу 3 Chat, задать вопрос.
4. Вернуться на Dashboard.
**Ожидание:** batch продолжает идти; в Chat агент работает с уже готовыми данными или сообщает что batch ещё идёт; на Weights слайдер сохранил положение через session_state.

### TC-E2E-031. F5 во время batch (P0)
**Шаги:**
1. Batch на 100 устройств.
2. При 50% — F5.
**Ожидание:**
- Singletons (factor_store, agent) переживают reload.
- UI показывает текущий статус batch.
- Batch продолжается с того места, где был.
- Нет дублирования HealthResult.

### TC-E2E-032. Перезапуск приложения (docker-compose restart) — потеря в памяти (P1)
**Шаги:** во время batch перезапустить сервис.
**Ожидание:**
- После рестарта session_state пустой (ожидаемо).
- Qdrant сохраняется (volume).
- Memory агента сохраняется (если реализовано персистентно).
- Пользователь видит «нет активных задач», может загрузить файл заново.

### TC-E2E-033. Две пользовательские сессии параллельно (P1)
**Шаги:** два браузера (разные профили), каждый делает batch.
**Ожидание:** сессии изолированы, не мешают друг другу.

## 5. Деградации и сбои

### TC-E2E-050. Реальная LLM возвращает частые ошибки / таймауты (P1)
**Шаги:** симулировать 30% failure rate на LLM.
**Ожидание:**
- Batch продолжает работать.
- Проблемные устройства помечены flag_for_review.
- Отчёт собирается, с индикацией проблем.

### TC-E2E-051. Qdrant временно недоступен (P1)
**Ожидание:** агент не может вызвать search_service_docs; severity ставится с флагом «unknown»; отчёт собирается с пометками.

### TC-E2E-052. Диск переполнен при генерации PDF (P1)
**Ожидание:** понятная ошибка; HTML доступен; повтор после освобождения места работает.

### TC-E2E-053. OOM при batch на слишком большом файле (P1)
**Ожидание:** либо graceful отказ до старта, либо частичный результат + ошибка.

### TC-E2E-054. Изменение weights во время batch (P2)
**Ожидание:** либо блокировка UI weights на время batch, либо warning «применится после завершения».
