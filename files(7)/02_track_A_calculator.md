# Тест-план 02 — Трек A: Calculator (формула индекса здоровья)

**Область:** `tools/calculator.py` — детерминированная формула H, confidence, зоны
**Владелец:** QA-инженер трека A
**Связанные документы:** `track_A_health_index_formula.md`, `architecture_overview.md` раздел 6.1, 6.2, 6.6

---

## Цель

Проверить, что формула индекса здоровья:
1. **Детерминирована** — одни и те же входы дают один и тот же выход, всегда.
2. **Корректна математически** — все факторы (S, R, C, A) считаются по спецификации.
3. **Устойчива к граничным случаям** — потолки, нули, отсутствие данных.
4. **Объяснима** — каждый вклад фактора трассируется до конкретного события.

Это **чистая функция** — ни LLM, ни сети, ни I/O. Тесты — unit-тесты высокой скорости, должны быть первыми зелёными.

## Контракты

- **Вход:** `list[Factor]` + `WeightsProfile`.
- **Выход:** `HealthResult` (frozen dataclass).
- **Инвариант:** `1 ≤ health_index ≤ 100`, `0.2 ≤ confidence ≤ 1.0`, `zone ∈ {green, yellow, red}`.

---

## 1. Детерминизм

### TC-A-001. Один и тот же вход → один и тот же выход (P0)
**Шаги:** вызвать `calculate()` 10 раз с идентичным списком `Factor`.
**Ожидание:** все 10 `HealthResult` идентичны побайтно (кроме `calculated_at`).

### TC-A-002. Порядок факторов не влияет на результат (P0)
**Шаги:** один и тот же набор, но в разном порядке.
**Ожидание:** `health_index` идентичен, `factor_contributions` — порядок может отличаться, но содержимое то же.

### TC-A-003. Отсутствие скрытого состояния (P0)
**Шаги:** вызвать calculate для device A, затем для device B, затем снова A.
**Ожидание:** результат для A оба раза одинаковый, не зависит от промежуточных вызовов.

### TC-A-004. Повторяемость на разных потоках (P1)
**Шаги:** параллельно вызвать calculate в 10 потоках с одинаковыми входами.
**Ожидание:** все 10 результатов идентичны.

## 2. Базовая формула (happy path)

### TC-A-010. Устройство без ошибок → индекс 100 (P0)
**Шаги:** `factors = []`, ресурсы в норме.
**Ожидание:** `health_index = 100`, `zone = green`, `confidence ≥ 0.9`.

### TC-A-011. Один minor фактор, 1 повтор (P0)
**Шаги:** S=минор, R=1, C=1.0, A=свежий.
**Ожидание:** индекс падает умеренно, остаётся в green; `factor_contributions` содержит 1 запись с корректным вкладом.

### TC-A-012. Один major фактор (P0)
**Ожидание:** индекс заметно ниже; yellow или граница yellow/red.

### TC-A-013. Один critical фактор (P0)
**Ожидание:** индекс значительно ниже; зависит от потолка C и модификаторов.

### TC-A-014. Смешанные factor'ы (1 critical + 2 major + 5 minor) (P0)
**Ожидание:** индекс считается корректно, соответствует ручному расчёту по формуле.

## 5. Фактор C (Component modifiers)

### TC-A-040. Модификатор применяется при попадании компонента в `applies_to_components` (P0)
**Шаги:** fault на fuser, правило для категории fusing со множителем 1.3.
**Ожидание:** C=1.3.

### TC-A-041. Модификатор не применяется вне scope (P0)
### TC-A-042. Несколько модификаторов (P1)
**Ожидание:** перемножаются, но не превышают потолок C.

### TC-A-043. **Потолок C ≤ 1.5** (риск 3 из архитектуры) (P0)
**Шаги:** подать factors с множителями 1.3, 1.4, 1.2 одновременно.
**Ожидание:** итоговый C ≤ 1.5 (clamping).

### TC-A-044. Кейс «1 Critical + C=1.5 × R=2» — индекс НЕ всегда 1 (P0)
**Шаги:** 1 critical ошибка с максимальным C и R.
**Ожидание:** по правилу из архитектуры «1 Critical в день не даёт индекс=1»; итоговый индекс не скатывается в минимум.

### TC-A-045. Модификатор по `model` (P1)
**Ожидание:** срабатывает для конкретной модели (напр., более хрупкая серия).

## Критерии приёмки трека A

- Все **P0** пройдены.
- Детерминизм подтверждён (TC-A-001, 002, 003).
- Reference-датасет (TC-A-130) даёт bit-exact совпадение.
- Потолки H (1..100), C (≤1.5), confidence (0.2..1.0) соблюдены.
- Риск 3 из архитектуры (Critical ×1.5×2 → всегда 1) закрыт тестами TC-A-043, TC-A-044.
- Формула изолирована: нет зависимостей от LLM, RAG, сети.
